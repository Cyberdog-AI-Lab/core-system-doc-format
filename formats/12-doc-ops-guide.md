# ドキュメント運用プロセスガイド（v0.1 たたき台）

**日付：** 2026-06-18
**記録者：** 羽床 ＋ Claude

> [補足] 基幹システム開発のドキュメント体系（12種）を、**いつ・誰が・どの順で作成／更新／レビュー／維持するか**を定める運用ガイド。**のちに skill に読み込ませ、この運用どおりにドキュメントが自動メンテされるよう制御する**ことを想定し、agent が follow できる手順・ルールとして書く。

---

## 1. 目的・適用範囲

- 対象：構想／用語集／要件／情報定義／機能仕様／API仕様／品質保証／システムアーキテクチャ／データ移行／システム維持保守ランブック／要件トレーサビリティ の各ドキュメント（＋本ガイド自身）。
- ゴール：**整合性（要件→成果物→検証）を保ったまま、ドキュメントが常に最新で・追跡可能**な状態を維持する。
- **運用主体：メンテナンスは AI（skill）を主体**とする。整合性を人手で保ち続けるのは破綻するため、変更の反映・トレーサビリティ更新・整合性チェックは AI が実行し、**人はレビューと承認に専念**する（詳細 §6・§10）。

## 2. ドキュメント体系（役割と軸）

| ドキュメント | 役割 | 軸 |
|--------------|------|----|
| 構想／コンセプト | 向きを揃える北極星 | 独立（チェーン外） |
| 用語集 | 語彙の唯一の源（用語・標準プロパティ名・値仕様） | 上流の共通基盤 |
| 要件 | 検証可能な単位で要望を管理 | 起点 |
| システムアーキテクチャ | 機能に依らない基盤・非機能の実現 | 基盤（物理の前提） |
| 情報定義 | 全仕様の源流（論理＋物理） | 機能：源流 |
| 機能仕様 | 各機能の詳細＋実装標準 | 機能 |
| API仕様 | I/F＋内部処理フロー | 機能 |
| 品質保証 | 検証記録の蓄積（保証の証跡） | 横の保証 |
| データ移行 | 移行の設計・実施記録 | データの引っ越し |
| システム維持保守ランブック | システム運用の手順（監視対応・障害・バックアップ・リリース）＋リリース台帳 | 運用 |
| 要件トレーサビリティ | 全改訂を要件駆動で一元記録 | 横断 |

## 3. 依存と整合性モデル

```
構想／コンセプト（独立・北極星）
   │ 向きを与える
   ▼
要件 ──（非機能）──▶ システムアーキテクチャ（基盤・物理の前提）
   │（機能）
   ▼
情報定義（源流）──▶［機能仕様 ──▶ API仕様］──▶ 実装（コード）
   │              （機能ドキュメント。両方が実装のインプット）  ▲
   └──▶ データ移行（随伴）      アーキも基盤として実装へ ──────┘

品質保証（横の保証）＝ 2系統で検証
  ・要件視点　：要件 R-xxx の受け入れ条件を保証
  ・成果物視点：情報定義 1-N／機能仕様 F-xxx／API仕様 API-xxx を保証
用語集＝語彙の唯一の源（全ドキュメントが参照）／要件トレーサビリティ＝全改訂を一元記録（横断）
システム維持保守ランブック＝システムを動かし続ける手順＋リリース台帳（運用。「リリース vX.Y」の唯一の定義元）
```

## 4. 作成タイミング（立ち上げ時の順序）

1. **構想／コンセプト**（最初・チーム全員で合意）／**用語集**（語彙の源を並行で立ち上げ、以降の全ドキュメントで参照・拡充）
2. **要件**（構想に沿って起票。機能／非機能）
3. **システムアーキテクチャ**（要件と並行・早め。情報定義の物理層・API整合性実装の前提）
4. **情報定義**（源流。論理 → 物理〔アーキ選定に依存〕）
5. **機能仕様 → API仕様**（情報定義から派生）
6. **実装**
7. **品質保証**（要件視点＋成果物視点）
8. **データ移行**（情報定義に基づき、初回＋リリースごと）
9. **システム維持保守ランブック**（リリース前に整備。以降、リリースのたびに台帳へ追記）
10. **要件トレーサビリティ**（全工程で随時更新）

## 5. 変更フロー（★ 変更時の標準手順）

> agent／チームは、ドキュメント変更時に必ずこの順で処理する。

1. **変更は原則「要件チケット」から起こす**（新規／変更／取り下げ）。誤記・リファクタ等は「要件外」として扱う。
2. **影響する下流ドキュメントを特定**（情報定義 → 機能仕様 → API仕様 → 実装 → 品質保証。必要に応じてアーキ・データ移行）。
3. **上流から順に反映**（情報定義 → 機能仕様 → API仕様 → 実装 → 品質保証）。**論理 → 物理**の順を守る。
4. **各反映を二か所に記録**：要件の**反映状況テーブル**（現況）＋**要件トレーサビリティ文書**（時系列）。箇所は ID で指す（1-N／F-xxx／API-xxx／QA-xxx）。
5. **品質保証で検証**し、品質保証の反映状況を「済」に更新。
6. **完了条件**：受け入れ条件すべて満たし、全層（品質保証を含む）が「済」。
7. **取り下げ時**：チケットを消さず、取り下げコンテキスト（理由・判断者・日付）を残す。

## 6. 担当・レビュー・承認（AI 主体・人は承認）

- **更新は AI（skill）が実行**し、各ドキュメントの**オーナー（人）がレビュー・承認**する。
- 変更は **AI 反映 → 人レビュー → 承認**で確定（承認状態はチケット／PR で管理）。
- 構想の改訂は**全員合意**を要する（独立だが影響大）。

## 7. メンテナンスの不変ルール（★ 守るべき制約）

1. **改訂履歴は各ドキュメントに持たず、要件トレーサビリティ文書に一元化**する。
2. **原本（元メモ・原文の記録）は移動・削除しない**（何を基に書いたかを保全する）。
3. **廃止・取り下げは消さず、状態で残す**（要件・情報定義・移行すべて。ID も欠番にしない）。
4. **用語は別文書「用語集」を唯一の源**とし、標準プロパティ名・値仕様を各ドキュメントで参照する（構想から切り出し済み）。
5. **値域・整合性・横断方針は「源」に1回だけ**書く（情報定義§3/§5、機能仕様§15 実装標準）。下流は**参照**し二重定義しない。
6. **特定プラットフォーム手段に依存しない**（概念で書き、製品はプロジェクトが埋める。物理層は論理から分離）。

## 8. 整合性レビュー観点（★ cross-doc チェック。consistency-check スキルの観点 A〜H と1:1）

変更後、以下が一致／追跡可能かを確認する：

- **A 値域の貫通**：**用語集の値仕様** ↔ 情報定義の値域 ↔ 機能仕様の入力チェック ↔ API仕様のバリデーション ↔ 品質保証のケース（源流は用語集）。
- **B 状態の貫通**：情報定義の値域（状態）↔ 機能仕様§12 状態遷移 ↔ 品質保証の不正遷移ケース。
- **C ID 連鎖の連続**：要件ID → 機能/画面ID → API ID → QA-ID が切れていない（宙ぶらりんのID・上流の無い"孤児"がない）。データ移行が参照する「リリース vX.Y」が、システム維持保守ランブックのリリース台帳（§7）に存在するか（台帳が唯一の定義元）。
- **D 反映の一致**：要件の反映状況テーブル／トレーサビリティのマトリクス ↔ 各ドキュメントの実体（箇所）。
- **E 横断方針の単一性**：横断的関心事が機能仕様§15 実装標準にのみ書かれている（各所に散っていない）。
- **F 用語集ドリフト**：標準プロパティ名・値仕様の表記ゆれ／未登録用語の混入／定義したのに使われていない用語。
- **G フォーマット準拠**：章一致・改訂履歴章を持たない・トレーサビリティのセルが正準形（状態 日付 反映先ID）。プラットフォーム固有の製品名が論理層に混入していないか（物理層は可）。
- **H スコープカバレッジ**：構想の「やること」に対応する要件が起票されている。

## 9. 粒度・線引きの指針

- **要件の粒度**：1チケット＝「受け入れ条件で検証できる」単位。大きすぎれば分割（親子＝関連要件）。
- **どこに書くか**：各フォーマットの「位置づけ」に従う。迷ったら「源に1回・下流は参照」。
- **どこまで書くか**：各章の目的を満たす最小限。網羅より追跡可能性を優先。
- **分冊のルール（基幹規模）**：情報定義・機能仕様が大きくなったら（目安：エンティティや画面が数十を超え、1ファイルで探すのがつらくなったら）、**業務領域（ドメイン）単位で分冊してよい**。その際：
  1. 親ファイルに**索引**（分冊の一覧とリンク）を置く。
  2. 章番号・ID（情報定義 1-N／F-xxx 等）は**全体で通し番号**を維持する（分冊ごとに1から振り直さない。トレーサビリティが壊れるため）。
  3. ダッシュボードには manifest.json の `docs` に分冊を1行足すだけでよい。

## 10. AI（skill）による維持（運用の主体）

ドキュメントのメンテナンスは原則 **AI（skill）が担い**、人はレビュー・承認に回る（§6）。AI は以下を実行する：

- 変更指示を受けたら **§5 変更フロー**に従い、上流 → 下流の順で反映し、**反映状況＋トレーサビリティを更新**する。
- **§8 整合性レビュー観点**を自動チェックし、不一致を指摘する。
- 改訂は**トレーサビリティ文書に記録**し、各ドキュメントには改訂履歴を作らない（§7-1）。
- **§7 不変ルール**を制約として常に守る。
