# 基幹システム開発 ドキュメント体系（仮決め）

> 基幹システム開発ドキュメントの記載フォーマット一式。**ここが正（single source of truth）**。
> 全12文書。各文書は**章一致構造・各章コピペ用スケルトン・プラットフォーム非依存**で統一。改訂履歴は各文書に持たず「要件トレーサビリティ」へ一元化。メンテナンスは**AI主体**（運用ガイド参照）。

---

## 整合性モデル

```
構想／コンセプト（独立・北極星）   用語集（語彙の唯一の源）
        │ 向きを与える                  │ 語彙・値仕様
        ▼                               ▼
      要件 ──（非機能）──▶ システムアーキテクチャ（基盤・物理の前提）
        │（機能）
        ▼
   情報定義（源流）──▶［機能仕様 ──▶ API仕様］──▶ 実装（コード）
        │              （機能ドキュメント。両方が実装のインプット）  ▲
        └──▶ データ移行（随伴）      アーキも基盤として実装へ ──────┘

   品質保証（横の保証）＝ 2系統で検証
     ・要件視点　：要件 R-xxx の受け入れ条件を保証
     ・成果物視点：情報定義 1-N／機能仕様 F-xxx／API仕様 API-xxx を保証

   要件トレーサビリティ ＝ 全改訂を要件駆動で一元記録（横断）
   ドキュメント運用プロセスガイド ＝ AI主体で維持・整合性を統制（横断）
   運用ランブック ＝ システムを動かし続ける手順＋リリース台帳（運用）
```

## フォーマット群を縦に貫く「貫通線」

| 貫通線 | 経路 |
|--------|------|
| **値域** | 用語集 → 情報定義 → 機能仕様 → API仕様 → 品質保証 |
| **状態** | 情報定義 → 機能仕様（状態遷移）→ 品質保証 |
| **トレーサビリティID** | 要件 → 機能 → API → 品質保証（QA） |
| **非機能** | 要件 → システムアーキテクチャ → 品質保証 |

> 「実装」はコード（ドキュメントではない）。整合性チェーンの最終成果物。
