# システム維持保守ランブック 記載フォーマット（v0.1 たたき台）

**日付：** 2026-07-01
**記録者：** 羽床 ＋ Claude

> [補足] システムを「作って終わり」にせず「動かし続ける」ための文書。監視ツールの選定はアーキ§5がやるが、**アラートが鳴ったあと何をするか**はここに書く。参考実物なし・第一原理。特定プラットフォーム手段には依存させない。
> ※ 名前が似ている「ドキュメント運用プロセスガイド（12）」は**文書の**運用、本書は**システムの**運用。別物。

---

## 位置づけ（整合性モデル）

- リリース後の**運用フェーズの受け皿**。体系の他文書が「どう作るか」を書くのに対し、本書は「どう動かし続けるか」を書く。
- **読者は運用担当**（開発者でない人も読む）。専門用語をなるべく避け、言葉は用語集に合わせる。手順は「誰がやっても同じ結果になる」粒度で書く。
- 他文書との関係：
  - アーキ§5（監視ツールの選定）→ 本書§3（鳴ったらどうするか）
  - アーキ§6（可用性の実現方針）→ 本書§5（バックアップ・リストアの手順）
  - 機能仕様§5（ジョブフロー）→ 本書§2（定常運用のカレンダー）
  - データ移行の「リリース vX.Y 随伴」→ 本書§7（**リリース台帳が vX.Y を定義する**）

## 全体構成（章立てテンプレート）

| 章 | 内容 |
|----|------|
| 1. 概要 | 目的・関連ドキュメント・連絡先の一覧 |
| 2. 定常運用 | 毎日・毎週・毎月やること（ジョブカレンダー） |
| 3. 監視・アラート対応 | アラートが鳴ったら何をするか |
| 4. 障害対応 | 障害レベルの定義・初動・エスカレーション・ふりかえり |
| 5. バックアップ・リストア | 取り方・戻し方・RPO/RTO・復旧訓練 |
| 6. リリース手順 | リリース前チェック・手順・切り戻し |
| 7. リリース台帳 | いつ・何を出したか（要件・移行への紐づけ） |

> 以下、各章の書き方とコピペ用スケルトンを章番号に沿って示す。改訂履歴は持たず、要件トレーサビリティ文書で一元管理する。

## 1. 概要

目的＋関連ドキュメント宣言＋**連絡先一覧**。連絡先は「役割名」で書き、実際の人名・電話番号は別表（またはチーム内の連絡先リスト）で管理する（人事異動のたびに本書を直さないため）。

```markdown
## 1. 概要
- 本書は {システム名} の運用手順（定常運用・アラート対応・障害対応・バックアップ・リリース）を定める。
- 関連ドキュメント：システムアーキテクチャ（§5 ツール・§6 可用性）／機能仕様（§5 ジョブフロー）／データ移行
- 連絡先（役割で書く）：
  | 役割 | 担当 | 連絡手段 |
  | ---- | ---- | -------- |
  | 一次対応（運用担当） | {別表参照} | {チャット/電話} |
  | 二次対応（開発担当） | {別表参照} | |
  | 最終判断（責任者） | {別表参照} | |
```

## 2. 定常運用

「毎日・毎週・毎月、誰が・いつ・何をするか」を1枚のカレンダーにする。ジョブ（バッチ）は機能仕様§5のジョブフローを参照し、**ここでは結果の確認方法だけ**書く（二重定義しない）。

```markdown
## 2. 定常運用
| 頻度 | 時刻 | やること | 担当（役割） | 確認方法 |
| ---- | ---- | -------- | ------------ | -------- |
| 毎日 | 09:00 | 夜間バッチ（機能仕様§5 B-xxx）の結果確認 | 一次対応 | {管理画面/ログの見方} |
| 毎週 | 月曜 | ディスク使用量の確認 | 一次対応 | {確認手順} |
| 毎月 | 1日 | {月次処理} | | |
```

## 3. 監視・アラート対応

**アラートごとに「意味」と「対応手順」をセットで**書く。対応の書けないアラートは作らない（鳴っても誰も動けないアラートはノイズになる）。

```markdown
## 3. 監視・アラート対応
| アラート | 意味（何が起きている） | 一次対応の手順 | 直らないとき（エスカレーション） |
| -------- | ---------------------- | -------------- | -------------------------------- |
| バッチ B-xxx 失敗 | 夜間処理が完了していない | 1. ログ確認 2. 再実行（冪等なので安全） | 再失敗なら二次対応（開発担当）へ |
| 応答遅延 | 画面表示が {N}秒 超過 | 1. 対象画面の特定 2. {確認手順} | 二次対応へ |
```

- **エスカレーション**＝一次対応で解決しないとき、誰に・いつまでに引き継ぐか。§1の役割表と対応させる。

## 4. 障害対応

障害の**レベルを先に定義**しておく（起きてから「これは重大か？」で揉めないため）。初動→復旧→ふりかえりの流れを固定する。

```markdown
## 4. 障害対応
### 4-1. 障害レベル
| レベル | 定義（例） | 目標復旧時間 | 通知範囲 |
| ------ | ---------- | ------------ | -------- |
| 重大 | 業務が止まる（例：登録ができない） | {N}時間 | 責任者まで即時 |
| 中 | 一部機能が使えない（回避策あり） | 当日中 | 開発担当まで |
| 軽微 | 表示くずれ等・業務は回る | 次リリース | チケット起票のみ |

### 4-2. 初動（どのレベルでも共通）
1. 事象の記録（いつ・何が・誰に影響）
2. レベル判定（4-1）→ 通知
3. 回避策の確認（{代替手段があれば}）
4. 復旧作業（必要なら §5 リストア・§6 切り戻し）

### 4-3. ふりかえり
- 復旧後に必ず記録する：原因／対応の時系列／再発防止（→ 必要なら要件チケット起票）
```

## 5. バックアップ・リストア

「何を・いつ・どこへ取り、**どう戻すか**」。バックアップは取っているだけでは意味がなく、**戻せて初めて意味がある**ので、リストア手順と訓練の記録を必須にする。

- **RPO**（どこまでのデータ喪失を許すか。例：24時間＝前日分までは諦める）
- **RTO**（どれだけの時間で復旧するか。例：4時間以内）

```markdown
## 5. バックアップ・リストア
- 取得対象・頻度：| 対象 | 頻度 | 保存先 | 保存期間 |
- RPO / RTO：{例 RPO=24時間 / RTO=4時間}（要件の可用性・アーキ§6と一致させる）
- リストア手順：1. {手順を誰でも実行できる粒度で} 2. ... 3. 復旧確認（{確認方法}）
- 復旧訓練の記録：| 実施日 | 実施者 | 所要時間 | 気づき |
```

## 6. リリース手順

リリースの**判定チェックリスト・手順・切り戻し**。**切り戻し手順の無いリリースはしない**。

```markdown
## 6. リリース手順
### 6-1. リリース前チェックリスト
- [ ] 含まれる要件の品質保証が済んでいる（要件の反映状況が全層「済」）
- [ ] 随伴するデータ移行（MIG-xxx）のリハーサルが済んでいる
- [ ] 切り戻し手順が準備できている（直前バックアップ＋旧版に戻す手順）
- [ ] 関係者へ告知済み（実施日時・影響範囲）

### 6-2. 手順
1. {実施時間帯：例 平日夜間} 2. バックアップ取得（§5） 3. デプロイ 4. データ移行（該当時） 5. 動作確認（{確認項目}） 6. 台帳に記録（§7）

### 6-3. 切り戻し
- 判断基準：{どうなったら戻すか} ／ 手順：{旧版に戻す具体手順}
```

## 7. リリース台帳

**「リリース vX.Y」を定義する唯一の場所**。データ移行や要件が「リリース v1.2 で」と参照する先は、この台帳の行。1リリース＝1行で、含んだ要件と随伴した移行を必ず紐づける。

```markdown
## 7. リリース台帳
| バージョン | 日付 | 含む要件 | 随伴移行 | 結果・メモ |
| ---------- | ---- | -------- | -------- | ---------- |
| v1.0 | 2026-07-15 | R-001〜R-005 | MIG-001 | 初回リリース |
| v1.1 | 2026-08-01 | R-006, R-008 | − | R-007 は見送り（取り下げ記録あり） |
```

## 記載ルール（章横断の作法）

1. **手順は「誰がやっても同じ結果になる」粒度で書く**（コマンド・画面名・確認方法まで。「適切に対応する」は手順ではない）。
2. **連絡先は役割名で書く**（人名・電話番号は別表で管理。人事異動で本書を腐らせない）。
3. **アラートは「意味」と「対応」をセットで**（対応を書けないアラートは作らない）。
4. **リリースは必ず台帳に1行残す**（含む要件・随伴移行への紐づけ必須。「vX.Y」の定義元は本書§7だけ）。
5. **切り戻し手順の無いリリースはしない**。
6. **リストア・障害対応は訓練し、実施日を記録する**（戻せないバックアップは無いのと同じ）。
7. **専門用語は用語集に合わせ、運用担当が読める言葉で書く**。
8. **改訂履歴は持たず、要件トレーサビリティ文書で一元管理する**。

## 貫通線（このフォーマット群との関係）

- **リリースの紐づけ**：リリース台帳（§7）が「vX.Y ↔ 含む要件 R-xxx ↔ 随伴移行 MIG-xxx」を結ぶ。データ移行の「リリース随伴」の参照先はここ。
- **非機能の貫通線（運用側の終点）**：要件の可用性・運用性 → アーキ§6（実現方針）→ **本書§5・§4（手順として実行可能に）**。
- **ジョブとの分担**：ジョブの定義（何を・いつ・依存）は機能仕様§5、**結果の確認と失敗時対応は本書§2・§3**（二重定義しない）。
