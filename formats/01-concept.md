# 構想／コンセプト 記載フォーマット（v0.1 たたき台）

**日付：** 2026-06-18
**記録者：** 羽床 ＋ Claude

> [補足] 参考にした実プロジェクト文書は無し。第一原理で設計。6分類のうち**唯一、整合性チェーンの外（独立）**にある文書。特定プラットフォーム手段には依存させない。

---

## 位置づけ（整合性モデル）

- 構想／コンセプトは整合性モデルの**チェーンの外（独立）**。要件 → 成果物 → 品質保証の直接の起点ではない。
- 役割は「具体的なシステム要件の前に、**全員の向くベクトルを合わせる**」こと。「どうしたいのか」が伝わり、チーム全員が**同じ絵**を描けるようにする。
- 直接トレースは持たないが、**要件が向かうべき北極星**。要件レビュー時に「この構想に沿うか」を確かめる。
- **短く・合意できる**ことを最優先（網羅より共感）。検証可能な詳細は要件以降に委ねる。

## 全体構成（章立てテンプレート）

| 章 | 内容 |
|----|------|
| 1. 概要 | このドキュメントの目的（向きを揃える）・読み方 |
| 2. 背景・課題 | なぜ今これをやるか・解決したいペイン |
| 3. ビジョンとコンセプト | 目指す姿・一言コンセプト・提供価値 |
| 4. 対象とステークホルダー | 誰のため・関係者とその期待 |
| 5. スコープの大枠 | やること／やらないこと（期待値調整） |
| 6. 成功の定義 | どうなったら成功か |
| 7. 制約・前提 | 予算・期間・体制・技術・法令の大枠 |> 全体で**1〜数ページ**を目安に。以下、各章の書き方とコピペ用スケルトンを示す。

---

## 1. 概要

このドキュメントの目的（関係者の向きを揃える）と読み方。詳細な要件・設計は別文書に委ねることを明記。

```markdown
## 1. 概要
- 本書は {プロダクト/プロジェクト名} の構想・コンセプトを定義し、関係者の向きを揃える。
- 方向性・価値・なぜ に絞る。検証可能な要件は「要件」、設計は情報定義以降で扱う。
```

## 2. 背景・課題

**なぜ今これをやるのか**。現状・解決したいペイン・機会（やらない場合のリスク）。

```markdown
## 2. 背景・課題
- 背景（なぜ今か）：
- 解決したい課題（現状のペイン）：
- 機会 / 放置した場合のリスク：
```

## 3. ビジョンとコンセプト

この文書の心臓。**目指す姿**（中長期）、**一言コンセプト**（覚えられる・エレベーターピッチ）、**提供価値**（誰の・何が・どう良くなる）。

```markdown
## 3. ビジョンとコンセプト
- ビジョン（目指す姿）：
- コンセプト（一言で）：{覚えられる1文}
- 提供価値：{誰の}{どんな課題が}{どう良くなる}
```

## 4. 対象とステークホルダー

**誰のためか**（主な対象ユーザー像）と、関係者およびその期待・関心。

```markdown
## 4. 対象とステークホルダー
- 主な対象（ユーザー像）：
- ステークホルダーと期待：
  | 関係者 | 期待・関心 |
```

## 5. スコープの大枠

**やること／やらないこと**を大枠で示し、期待値を揃える。詳細スコープは要件で詰める。

```markdown
## 5. スコープの大枠
- やること（この構想に含む）：
- やらないこと（含まない・将来検討）：
```

## 6. 成功の定義

**どうなったら成功か**。定性で良いが「判断できる形（〜できている状態）」に。あれば成功指標も。

```markdown
## 6. 成功の定義
- どうなったら成功か（定性）：「〜できている状態」
- （あれば）成功指標：
```

## 7. 制約・前提

大枠の**制約**（予算・期間・体制・技術・法令など）と、**前提**（崩れると構想が成り立たない仮定）。

```markdown
## 7. 制約・前提
- 制約：予算 / 期間 / 体制 / 技術 / 法令 など
- 前提（成り立たないと崩れる仮定）：
```

---

## 記載ルール（章横断の作法）

1. **短く、読めば向きが分かる**ことを最優先（網羅より共感・合意）。1〜数ページ。
2. **検証可能な要件は書かない** — それは「要件」フォーマットへ。ここは方向性・価値・なぜ・どうしたい。
3. **コンセプトは一言で言い切る**（覚えられる粒度／エレベーターピッチ）。
4. **やらないことを必ず書く**（期待値を揃え、暴走を防ぐ）。
5. **成功の定義は「判断できる形」に**（定性でも「〜できている状態」と言い切る）。
6. **用語は別文書「用語集」で一元管理する**（構想から切り出し済み。構想では用語の意味を厳密に詰めず、用語集に委ねる）。
7. **チーム全員で合意**し、変えたら要件トレーサビリティ文書に改訂を記録する（個別の改訂履歴は持たない。独立だが生きた文書）。
8. **整合性チェーンの外** — 成果物への反映トレースは持たない。ただし**要件はこの構想に沿うべき**（北極星）。

## 貫通線（このフォーマット群との関係）

- **位置づけ**：整合性チェーンの**外（独立）**。要件→成果物→品質保証の起点ではないが、**要件が向かう北極星**。要件レビューで「構想に沿うか」を確認する。
- **用語集との関係**：語彙は別文書「用語集」が唯一の源（構想から切り出し済み）。構想は「向き・価値・なぜ」に専念する。
- **スコープの大枠 → 要件のスコープ**：大枠の「やる/やらない」が、個々の要件チケットのスコープ判断の拠り所になる。
