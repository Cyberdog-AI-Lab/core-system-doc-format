# 文書一式の生成（doc-scaffold）スキル

テンプレート（`oss/core-system-doc-format/`）から、新しいプロジェクトの**動くドキュメント環境**を一式生成する。
図書貸出インスタンス（`instance/library-lending/`）を手作業で作ったときの手順を自動化したもの。

## 進め方

### 1. 最小ヒアリング（3問だけ）

1. **システム名**（例：図書貸出システム）→ 各文書の `{システム名}` に差し込む
2. **ドメイン概要を3行**（何を・誰が・どうするシステムか）→ 構想の下書きに使う
3. **DBの方向**（RDB／NoSQL／未定）→ 情報定義の物理章の案内に使う（未定なら論理層から書き始めてよい旨を書く）

これ以上は聞かない。**内容は創作しない**（埋めるのはプロジェクトの人間とAIの共同作業。scaffold は器を作るだけ）。

### 2. フォルダ構成の生成

出力先（ユーザー指定。例 `instance/{project-slug}/` や別リポジトリ）に：

```
{project}/
├── docs/               ← 12文書のスケルトン（★これから埋める本体）
├── formats/            ← 記載フォーマット一式のコピー（このプロジェクトの標準）
├── dashboard/          ← 俯瞰ダッシュボード（index.html＋manifest.json）
├── consistency-check/  ← 整合性チェックskill のコピー
├── change-propagate/   ← 変更反映skill のコピー
├── CLAUDE.md           ← AI向けの運用指示（instance/library-lending/CLAUDE.md を基にシステム名を調整）
└── README.md           ← 構成と起動方法
```

### 3. docs/ のスケルトン生成（12文書）

各フォーマット定義の末尾にある「**コピペ用・空スケルトン**」を使い、`{システム名}` を差し込んで
`docs/01-concept.md` 〜 `docs/12-ops-runbook.md` を生成する。

- 構想（01）だけは、ヒアリングの3行を「2. 背景・課題」の下書きとして入れてよい（`> [下書き]` と明示）。
- トレーサビリティ（10）の §2 マトリクスは**ヘッダ行だけ**置く（要件が起票されたら行が増えていく）。
  セルの書き方（正準形「状態 日付 反映先ID」・未は`未`・対象外は`−`）の説明コメントを1行添える。
- 運用ランブック（12）のリリース台帳もヘッダだけ置く。

### 4. dashboard の設置

- テンプレの `dashboard/index.html` をそのままコピー（**編集しない**。全プロジェクト共通）。
- `dashboard/manifest.json` を生成して差を吸収する：

```json
{
  "pageTitle": "{システム名} ドキュメントダッシュボード",
  "title": "{システム名} — ドキュメント",
  "subtitle": "実文書12本を俯瞰・進捗追跡・閲覧（docs/ から動的読込）",
  "docsDir": "../docs/",
  "instance": { "name": "{システム名}", "file": "../docs/10-traceability.md" },
  "docs": [ …テンプレの manifest.json の docs をそのままコピー… ],
  "aliases": [ …同上… ]
}
```

- 起動方法を README に書く：`npx serve {project} -p {port}` → `/dashboard/`。

### 5. 完了報告と次の一歩の案内

生成したファイル一覧を報告し、**書き始める順序**（運用ガイド §4）を案内する：

> まず 01構想（チームで合意）と 02用語集 → 03要件を起票 → 04アーキ → 05情報定義 → …
> 以後の変更は change-propagate、整合の点検は consistency-check で。

## 守るルール

- **内容を創作しない**。スケルトンは空欄のまま渡す（ヒアリングの3行だけ `> [下書き]` として入れてよい）。
- フォーマット（formats/）は**変更せずコピー**する。プロジェクト固有の都合はまず docs/ 側で吸収し、フォーマット自体を変えたい場合はテンプレート側への提案として報告する。
- dashboard/index.html は編集しない（差は manifest.json で吸収）。
- 既存フォルダへの上書きはしない（生成先が空でなければ確認する）。
