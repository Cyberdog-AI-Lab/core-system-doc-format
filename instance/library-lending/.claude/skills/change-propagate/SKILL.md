# 変更反映（change-propagate）スキル

運用ガイド §5「変更フロー」の実行体。**変更を受け付け、影響する文書を特定し、上流→下流の順に反映し、記録を残す**。
反映のたびに人がレビューできるよう、**既定は1層ずつ提案→確認**。「一括で反映して」と言われたときだけ全層を適用し、最後にまとめて報告する。

## 前提：反映の順序（これだけは崩さない）

```
要件 → 情報定義 → 機能仕様 → API仕様 → 実装 → 品質保証
        （非機能要件は 要件 → アーキテクチャ → 実装 → { 品質保証（検証）／ システム維持保守ランブック（運用手順・リリース台帳） }）
```

- 上流を直してから下流。**論理→物理**の順。
- 語彙・値仕様を変えるときは、まず**用語集**（唯一の源）から。

## 進め方

### 1. 変更の受付（入口はいつも要件）

- 変更内容を聞き、**対応する要件チケット（R-xxx / NFR-xxx）を特定**する。
  - 該当チケットが無い → 新しいチケットの**起票を提案**する（要件フォーマットのスケルトンで下書き）。
  - 誤記修正・リファクタなど要件に紐づかない変更 → トレーサビリティ文書の「**要件外の改訂ログ**」行きとして扱う（チケットは作らない）。
- 取り下げ（却下・廃止・保留）の場合：チケットは消さず、**取り下げコンテキスト**（理由・判断者・日付・代替）を書く。

### 2. 影響する文書の特定

変更が届くべき層を洗い出し、ユーザーに一覧で見せる：

```
反映計画：R-012「返却期限の延長申請」
  1. 用語集      … 「延長」の用語・値仕様を追加     ← 変更あり
  2. 情報定義    … 貸出エンティティに延長回数を追加  ← 変更あり
  3. 機能仕様    … F-003 返却画面に延長ボタン        ← 変更あり
  4. API仕様     … API-005 延長APIを新設             ← 変更あり
  5. アーキテクチャ … −（対象外）
  6. 実装        … （コード側。文書は対象外）
  7. 品質保証    … QA-xxx 延長の検証ケース（実施後に記録）
  8. システム維持保守ランブック … −（この例では対象外）
```

- 対象外の層は「−」と明示する（あとで反映状況にもそのまま書く）。
- **バッチ・通知・非機能（可用性など）に触る変更は、システム維持保守ランブックへの影響も確認する**（定常運用のカレンダー・アラート対応・リリース台帳）。

### 3. 上流から順に反映（1層ずつ）

各層で：**変更案を提示 → ユーザー確認 → 適用**。適用したら次の層へ。

- 書き方は各フォーマット定義（`formats/`）に従う。章構成を崩さない。
- 値域・プロパティ名は**用語集を源**に。下流には再定義せず参照を書く。
- 廃止・削除は消さず「廃止（日付）」で残す。ID は欠番にしない。
- 「一括で反映して」と言われた場合のみ、全層を続けて適用し、最後に差分の要約を報告する。

### 4. 記録の更新（反映のたびに2か所）

1層反映するごとに、必ず記録する：

- **トレーサビリティ文書 §2 マトリクス**：該当セルを更新。書式は正準形「**状態 日付 反映先ID**」（例 `済 2026-07-01 F-003`）。状態は `未`／`反映中`／`済` の3語だけ。
- **トレーサビリティ文書 §3 タイムライン**：1反映＝1行（日付｜対象ドキュメント｜箇所｜変更内容｜版/コミット｜実施者）。
- 要件チケット側に反映状況テーブルがある場合はそれも同じ状態に揃える。

> 各文書に改訂履歴は書かない（運用ガイド §7-1）。記録はトレーサビリティ文書だけ。

### 5. 仕上げの整合確認

反映が済んだら、変更に関係する範囲だけ consistency-check の観点で自己点検する：

- 値域を変えた → A（値域の貫通）：用語集→情報定義→機能→API→QA が一致しているか
- 状態を変えた → B（状態の貫通）：状態遷移表と QA ケース
- ID を足した／リリースに紐づけた → C（ID連鎖）：宙ぶらりんの ID が無いか。データ移行が参照する「リリース vX.Y」がランブックの台帳に実在するか
- 常に → D（反映状況↔実体）：マトリクスと実物の記述が一致しているか

ズレを見つけたら、その場で直すか、指摘として報告する。

### 6. 完了報告

- 反映した層と箇所（ID）・更新したマトリクスのセル・タイムラインの追記行を一覧で報告する。
- 品質保証が未実施なら「QA が残っている」ことを明示する（済にしない）。

## 守るルール（運用ガイド §7）

- 変更は**要件起点**。要件外はすべて「要件外の改訂ログ」へ。
- 上流→下流・論理→物理の順を守る。飛ばして下流だけ直さない。
- 改訂履歴は各文書に書かず、**トレーサビリティ文書へ**。
- **取り下げ・廃止は消さない**（状態と経緯で残す）。
- 用語集が語彙の唯一の源。値仕様の直書きを増やさない。
- プラットフォーム固有の書き方は物理層（アーキ§3・情報定義§5）だけに。
