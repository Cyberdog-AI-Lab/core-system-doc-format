# 整合性チェック スキル

ドキュメント体系の**文書間の整合性を検査し、不整合をレポート**する。既定は **read-only（検査のみ）**。修正は明示指示があるときだけ行う。

## 前提：対象ドキュメント体系

整合性チェーン：**要件 →（情報定義 → 機能仕様 → API仕様 → 実装）→ 品質保証**。上流に構想・用語集、横断にトレーサビリティ・運用ガイド、基盤にシステムアーキテクチャ、随伴にデータ移行、運用に運用ランブック（リリース台帳）。

- 用語集 … 用語・標準プロパティ名・値仕様（型/桁数/利用可能文字/値域）の唯一の源
- 情報定義 … エンティティ・値域（§1）・業務ルール（§3）・バリデーション（§3-2）・物理（§4-5）
- 機能仕様 … 画面/バッチ等、状態遷移（§12）、実装標準（§15）、ジョブフロー（§5）
- API仕様 … I/F・内部処理フロー・エラーコード
- 品質保証 … 検証記録（対象＝成果物の箇所、観点、何が保証できたか）
- 要件 … 受け入れ条件・反映状況テーブル（情報定義/機能/API/実装/品質保証）
- 要件トレーサビリティ … 全改訂の時系列＋マトリクス（セル正準形「状態 日付 反映先ID」）
- 運用ランブック … 定常運用・アラート対応・障害・バックアップ・リリース手順。§7 リリース台帳が「リリース vX.Y」の唯一の定義元

> 文書名・ファイル名はプロジェクトで異なりうる。役割で同定する。同梱の `formats/`（フォーマット定義）と運用ガイド（11）を正本として参照する。

## 進め方

1. **対象の決定**：ユーザーが対象ディレクトリを指定したらそれを検査する（例「instance/library-lending/docs をチェックして」）。未指定なら、現在のプロジェクトから体系の文書（要件・情報定義…）を役割で探して確認を取る。
2. **対象を収集**：体系の各文書を読み込む（無い文書はスキップし、レポートに「未作成」と記す）。
3. **観点 A〜H を検査**（下記）。各指摘に **該当箇所（文書・章・ID）** と **重要度（🔴高/🟡中/🟢低）** を付す。
4. **レポート出力**（末尾テンプレ）。既定は**チャットに報告**。「ファイルに残して」と言われたら対象ディレクトリに `consistency-report-{日付}.md` として保存する。
5. 既定は report-only。「修正して」と言われた場合のみ、**上流→下流（論理→物理）**の順で直し、**要件トレーサビリティに改訂を記録**する（change-propagate と同じ作法）。

## 検査観点

### A. 値域の貫通（最重要）
**用語集の値仕様 → 情報定義の値域 → 機能仕様の入力チェック → API仕様のバリデーション → 品質保証のケース** が一致・参照で繋がっているか。
- 用語集に無い値仕様が下流に直書きされていないか（源に1回・下流は参照）。
- 同じ属性の値域が、情報定義と API バリデーションで食い違っていないか。
- 値域があるのに品質保証に境界値/列挙外のケースが無い（未検証）箇所。

### B. 状態の貫通
**情報定義の値域（状態）→ 機能仕様§12 状態遷移 → 品質保証の不正遷移ケース**。
- 情報定義で定義した状態が、機能仕様の状態遷移表で網羅されているか。
- 不正遷移を拒否する品質保証ケースがあるか。

### C. トレーサビリティID連鎖
**要件ID → 機能/画面ID → API ID → QA-ID** が切れていないか。
- 要件の反映状況テーブルの「反映先ID」が**実在**するか（宙ぶらりんのID＝デッドリンク）。
- 下流（機能/API/QA）に、紐づく上流IDの無い"孤児"が無いか。
- データ移行が参照する「リリース vX.Y」が、運用ランブックの**リリース台帳に存在**するか（台帳が唯一の定義元）。

### D. 反映状況 ↔ 実体
要件の反映状況テーブルと、各文書の実体が一致するか。
- 「反映済」とあるのに、対象文書に該当の記述が**無い**（虚偽の反映済）。
- 文書に変更があるのに、反映状況／トレーサビリティに**記録が無い**。
- 取り下げ（却下/廃止/保留）要件が、下流にまだ"反映済"として残っていないか。

### E. 横断単一性
**横断的関心事**（命名・エラー・ログ・日時/通貨・i18n・トランザクション・セキュリティ・設定・静的解析・バッチ運用）が、**機能仕様§15 実装標準にのみ**集約され、各機能/各APIに散っていないか。

### F. 用語集ドリフト
**標準プロパティ名・値仕様**が、情報定義の属性・API の項目名で用語集と一致しているか。表記ゆれ・別名（正式名称を使っているか）・未登録用語の混入。
**逆向きも見る**：用語集で定義したのに、下流のどの文書でも使われていない**未使用の用語・コード値**（定義だけが先行して腐っていく前兆）。

### G. フォーマット準拠
各文書が**章一致構造**で、**改訂履歴章を持たない**（→トレーサビリティに集約）、必須章・コピペ用スケルトン項目が埋まっているか。プラットフォーム固有の製品名が**論理層に混入**していないか（物理層は可）。
トレーサビリティ§2マトリクスのセルが正準形「**状態 日付 反映先ID**」（状態＝未/反映中/済・対象外＝−）で書かれているか（崩れているとダッシュボードで進捗が見えなくなる）。

### H. スコープ↔要件カバレッジ
**構想の「やること」に挙がった項目に、対応する要件チケットがあるか**。
- 構想§5（スコープの大枠）の各項目 → 要件（R-xxx / NFR-xxx）が起票されているか。無ければ「スコープにあるのに要件が無い」抜け。
- 逆に、下流（エンティティ・状態・機能・バッチ）にあるのに要件が無い"孤児"は C観点で拾う。両方合わせて「入口と出口の抜け」を塞ぐ。

## テンプレ自己検査モード

「**テンプレ自身をチェックして**」と言われたら、検査対象を実プロジェクトの文書ではなく**フォーマット定義一式（formats/・スキル・ダッシュボード設定）**に切り替え、テンプレ内部のドリフトを見る：

- **文書数の一致**：formats/ のファイル数 ↔ formats/README の一覧 ↔ 運用ガイド§1・§2 の一覧 ↔ ダッシュボード manifest の docs 数。
- **貫通線の定義一致**：README の貫通線表 ↔ 各フォーマットの「貫通線」節 ↔ 本スキルの観点（例：値域の源流が「用語集」で揃っているか）。
- **観点の対応**：運用ガイド§8 のレビュー観点 ↔ 本スキルの A〜H（片方だけ増えていないか）。
- **相互参照の生存**：フォーマット間の「§n参照」が、参照先の実際の章立てに存在するか。

> 実例：2026-07-01 の評価で「ガイド冒頭が9種のまま」「§8の値域貫通に用語集が無い」等のドリフトを検出した。テンプレも文書である以上、同じように腐る。

## 守るルール（運用ガイド §7）

- **原本（元メモ・原文の記録）を勝手に移動・削除しない**。
- 修正する場合も**改訂履歴は各文書に書かず、要件トレーサビリティへ**記録する。
- **用語集を唯一の源**とし、値域・標準プロパティ名はそこに合わせる。
- **廃止・取り下げは消さず**状態で残す。
- 検査は中立に。**事実（不整合）・推測・不明点を分けて**報告する。
