# 図書貸出システム — ドキュメント・インスタンス（検証用プロジェクト）

`oss/core-system-doc-format`（＝テンプレート）を**コピーして実際に埋めた、動くサンプルプロジェクト**。
「基幹システム開発のドキュメント体系」がひととおり機能するかを検証するための一式です。

> 題材：公共図書館の貸出システム（書誌・蔵書・利用者・貸出・予約・延滞）。

## 構成

```
library-lending/
├── docs/               ← ★このプロジェクトの実文書（フォーマットに沿って記述した12本）
├── formats/            ← 採用した記載フォーマット（テンプレートからのコピー＝標準）
├── dashboard/          ← 俯瞰ダッシュボード（index.html は共通・manifest.json が docs/ を指す）
├── consistency-check/  ← 整合性チェックskill（検査対象＝docs/）
├── change-propagate/   ← 変更反映skill（要件→下流への反映とトレサビ更新）
└── README.md           ← 本ファイル
```

- **正（single source of truth）はテンプレート側** `oss/core-system-doc-format/`。ここはその**写し＋実データ**。
- `docs/` が「埋めた実文書」。`formats/` はその文書が従う標準（テンプレのコピー）。

## docs/ の12文書

要件ID（R-001〜R-005）を起点に、値域・状態・トレーサビリティID・非機能の**4本の貫通線**が全文書を貫くよう記述してある。

| # | 文書 | 図書貸出での中身（例） |
| --- | --- | --- |
| 01 | 構想 | 二重貸出ゼロ・照会1秒・延滞自動検出をNorth Starに |
| 02 | 用語集 | `CopyStatus`/`MemberType`、上限冊数（一般5/学生10）などの**値域の源** |
| 03 | 要件 | R-001〜R-005＋非機能NFR-01〜03 |
| 04 | アーキ | 論理ロール＋物理採用（RDB）・ADR-001（強整合で二重貸出防止） |
| 05 | 情報定義 | 書誌/蔵書/利用者/貸出/予約・「蔵書＋未返却」一意制約 |
| 06 | 機能仕様 | F-001〜F-004・延滞検出バッチB-001・状態遷移・実装標準 |
| 07 | API仕様 | API-001〜004（F-xxxの外部I/F） |
| 08 | 品質保証 | QA-001（実施済）〜QA-005（計画） |
| 09 | データ移行 | MIG-001 旧台帳CSV→各エンティティ |
| 10 | トレーサビリティ | **要件×文書のマトリクス**（セル＝「状態 日付 反映先ID」。ダッシュボードのタイムライン駆動データ） |
| 11 | 運用ガイド | 本プロジェクトへの適用（AI主体メンテ・不変ルール） |
| 12 | 運用ランブック | 定常運用・アラート対応・障害・バックアップ（RPO/RTO）・リリース手順＋リリース台帳（v1.0） |

## 動かす

```bash
# リポジトリ直下で（プレビュー名: doc-dashboard-lib / port 4323）
npx serve oss/core-system-doc-format/instance/library-lending -p 4323
# → http://localhost:4323/dashboard/
```

- **俯瞰**：整合性モデル＋11文書カード。ボックス→該当実文書へ。
- **トレーサビリティ**：R-001〜R-005 の進捗タイムライン（`docs/10-traceability.md` を読む）。
- **ドキュメント**：docs/ の実文書を目次・参照チップつきで閲覧。

## 整合性チェック（skillの検証）

`consistency-check` skill を **このプロジェクトの `docs/` に対して**走らせると、
値域貫通・状態貫通・トレースID連鎖・反映状況↔実体・用語集ドリフトなどを検査できる。
＝テンプレートのフォーマット群が「本当に整合を検出できる形になっているか」を実データで検算する場。
