**日付：** 2026-06-16 作成（v0.1＝第一原理 ／ v0.2＝全体スケルトン整備）／ 2026-06-18 v0.3＝実プロジェクトの API 仕様をベースに章一致構造で再構成（プラットフォーム非依存で記述）
**記録者：** 羽床 ＋ Claude

> [補足] 実在の予約系基幹システムの API 詳細仕様書（約3100行）を元に、章立て・型定義・記載作法を抽出して汎用テンプレート化したもの。**特定のプラットフォーム手段（ランタイム・DBの API 名・通信プロトコル固有の挙動）には依存させず、概念で記述する**。情報定義（05）・機能仕様（06）と対で使う。

---

## 位置づけ（整合性モデル）

- **情報定義 → 機能仕様 →【API仕様】→ 実装** の縦チェーンの3段目。機能仕様が定めた操作を、**I/F（型）＋内部処理フロー**として確定し、実装の直接の元にする。
- 各APIに**利用画面**（機能 → API の逆引き）と**関連情報定義**（情報定義 → API）を必ず付ける＝トレーサビリティ。
- ここで**整合性の担保**（トランザクション境界・楽観ロック）を確定する＝情報定義§4「整合性の担保場所」の実装側。

## 全体構成（章立てテンプレート）

| 章 | 内容 | 備考 |
|----|------|------|
| 1. 概要 | 目的・関連ドキュメント（機能仕様/情報定義）への参照 | |
| 2. 共通仕様 | 認証認可・共通レスポンス形式・エラー方針・エラーコード一覧・共通型定義 | |
| 3〜. API詳細（カテゴリ別） | カテゴリごとに章を分け、各APIを定型テンプレで記載 | ドメイン依存 |
| マスタAPIの共通パターン | CRUD（list/get/create/update/delete）を1回定義し、マスタ別は差分のみ | |
> API群はドメインに応じて**カテゴリ別に章立て**する（例：予約／船割当／支払／マスタ／システム管理）。以下、各章の書き方とコピペ用スケルトンを示す。
> ※ API → 実装コードの対応表は「実装」層（次の文書）の責務とし、本書には含めない。

## 1. 概要

目的＋**関連ドキュメント宣言**（機能仕様・情報定義への参照）。「各APIのI/Fと内部処理フローを定義する」ことを明記。

```markdown
## 1. 概要
- 本書は {システム名} の API 詳細仕様。各APIの I/F と内部処理フローを定義する。
- 関連ドキュメント：機能仕様（functional_spec）／情報定義（info_def）
```

## 2. 共通仕様

全API横断の取り決めを**冒頭に1回**まとめ、各APIでは再記述しない（情報定義の「共通項目外出し」と同じ思想）。

- **認証・認可**：認証基盤と、ロールベースの認可方式
- **共通レスポンス形式**：成功 `{ data }` ／ エラー `{ errors: [{ errorType, message, errorInfo? }] }`
- **エラーの返し方の方針**：トランスポート層の成否（HTTPステータス等）とビジネスエラーの表現方法を、採用プロトコルに合わせて定める。**クライアントのエラー判定方法を統一する**（ステータスで見るか／レスポンスボディのエラー領域で見るか）。プロトコル固有の挙動はプロジェクトで具体化する。
- **エラーメッセージの言語**：ユーザー表示用 `message` は日本語で返す
- **エラーコード一覧**：`VALIDATION_ERROR` / `NOT_FOUND` / `CONFLICT` / `BUSINESS_RULE_VIOLATION` / `UNAUTHORIZED` / `FORBIDDEN` / `INTERNAL_ERROR`
- **共通型定義**：ページネーション・日付/時刻型・金額型などを共通化

```typescript
// 共通型（コピペ用・型表記は言語に合わせる。例として TypeScript）
interface PaginationInput { limit?: number; nextToken?: string; }
interface PaginatedResult<T> { items: T[]; nextToken?: string; totalCount?: number; }
type DateString = string;     // YYYY-MM-DD
type DateTimeString = string; // ISO8601
type Money = number;          // 整数・円（原則税込）
type Rate = number;           // 0〜1の小数
```

**コピペ用スケルトン（共通仕様の章）**

```markdown
## 2. 共通仕様
### 2-1. 認証・認可：{認証基盤}／{ロールベース認可}
### 2-2. 共通レスポンス形式：成功 { data } ／ エラー { errors: [{ errorType, message, errorInfo? }] }
### 2-3. エラーの返し方：トランスポート成否とビジネスエラーの表現／クライアントの判定方法を統一
### 2-4. メッセージ言語：ユーザー表示用は日本語
### 2-5. エラーコード一覧：| コード | 説明 |
### 2-6. 共通型定義：Pagination / DateString / Money / Rate ...
```

## 3. API詳細のテンプレート（カテゴリ別）

実際の文書では API を**カテゴリ別に章立て**（予約／船割当／支払／…）。各APIは次の定型セクションで書く。

| セクション | 書き方 |
|-----------|--------|
| 概要 | 1文 |
| **利用画面** | 機能仕様の画面（機能 → API の逆引き） |
| **関連情報定義** | 情報定義の項番 |
| 認可 | 特殊権限のAPIのみ明記（例：ADMIN限定） |
| リクエスト | **型定義**（言語の interface 等）。更新系は**楽観ロック version 必須** |
| レスポンス | **型定義** |
| 内部処理フロー | 番号つき手順（下記の定型骨格） |
| バリデーションエラー | コード｜条件｜メッセージ（→ 品質保証異常系と 1:1） |
| 監査ログ（任意） | 監査用ログに残す操作種別（operation） |

**内部処理フローの定型骨格**

1. 入力バリデーション（必須・形式・存在）
2. マスタ存在・有効期間チェック（情報定義の該当エンティティ）
3. 業務ルールチェック（**情報定義§3の番号で参照**：定員・貸切・期日 等）
4. データ操作（**トランザクション境界**：原子的にまとめる範囲／一括処理の件数上限、**楽観ロック**：version による条件付き更新）
5. レスポンス構築（ページング・ソート。最終キー＝作成日時降順）

**コピペ用スケルトン（API 1本）**

```markdown
### {API名}
- 概要：
- 利用画面：{機能仕様の画面}
- 関連情報定義：1-N, 3-N
- 認可：{特殊権限があれば}
- リクエスト：
  ```
  interface {Api}Input { /* ... version: number（更新系） */ }
  ```
- レスポンス：
  ```
  interface {Api}Output { /* ... */ }
  ```
- 内部処理フロー：
  1. 入力バリデーション
  2. マスタ存在・有効期間チェック
  3. 業務ルールチェック（情報定義 3-x）
  4. データ操作（トランザクション境界 / version楽観ロック）
  5. レスポンス構築
- バリデーションエラー：| コード | 条件 | メッセージ |
```

## 4. マスタAPIの共通パターン

マスタ系は **CRUD（list / get / create / update / delete）の I/F を1回だけ共通定義**し、マスタ別は「型定義＋関連情報定義＋差分（固有フィルタ・固有項目・**削除制約**）」だけ書く。

```markdown
#### 共通パターン
- list{Master}s：{ フィルタ?, pagination? } → { items[], nextToken? }（デフォルトソート明記）
- get{Master}：{ id } → { item }
- create{Master}：{ 固有項目 } → { item }
- update{Master}：{ id, version, 固有項目 } → { item }   ← 楽観ロック
- delete{Master}：{ id } → { success }                   ← 参照されていれば削除不可

#### {マスタ名}API（関連情報定義：1-N）
- 型定義：interface {Master} { ... version; createdAt; updatedAt; }
- 差分：固有フィルタ／固有項目／削除制約（参照チェック）
```

## 記載ルール（章横断の作法）

1. **共通仕様を冒頭に1回**（認証・レスポンス形式・エラーコード・共通型）。各APIで再記述しない。
2. **各APIに「利用画面」と「関連情報定義」を必ず付ける** — 機能 → API、情報定義 → API の二重リンク（トレーサビリティ）。
3. **I/F は型で厳密に定義**（リクエスト／レスポンスとも）。曖昧さを残さない。
4. **内部処理フローは定型骨格で書く** — 入力検証 → マスタ存在 → 業務ルール（情報定義§3参照）→ データ操作 → レスポンス構築。
5. **整合性の実装を明記** — トランザクション境界（原子的にまとめる範囲）・楽観ロック（version による条件付き更新）・一括処理の上限・冪等性。＝情報定義§4「整合性の担保場所」の実装。
6. **更新系は楽観ロック version をリクエスト必須**にする。
7. **エラーは共通コード＋日本語メッセージ**。APIごとに（コード｜条件｜メッセージ）表を置く → そのまま品質保証の異常系ケース。
8. **業務ルール・計算は情報定義§3を番号で参照**（再記述しない）。
9. **マスタ系は CRUD 共通パターンを1回定義**し、マスタ別は差分のみ。
10. **改訂履歴は持たず、要件トレーサビリティ文書で一元管理する**／（任意）監査ログの操作種別を載せる。
11. **特定プラットフォーム手段に依存しない** — ランタイム・DBの API 名・プロトコル固有挙動は「概念」で書き、具体は採用技術ごとにプロジェクト側で補う。

## 貫通線（このフォーマット群を縦に貫くもの）

- **値域の貫通線**：情報定義の値域 →（機能仕様）入力チェック →（API仕様）リクエストの**型・必須・バリデーション（§3参照）** → 品質保証のバリデーションケース。
- **整合性の貫通線**：情報定義§4「整合性の担保場所」 →（API仕様）内部処理フローの**トランザクション境界・条件付き更新（楽観ロック）**で実体化 → 品質保証の整合性ケース。
- **トレーサビリティID連鎖**：要件 → 機能（利用画面/機能ID）→ **API（本書）** → 品質保証ID。
