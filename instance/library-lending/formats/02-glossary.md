# 用語集 記載フォーマット（v0.1 たたき台）

**日付：** 2026-06-18
**記録者：** 羽床 ＋ Claude

> [補足] システム全体の**語彙の唯一の源（ユビキタス言語＋データ辞書）**。各業務用語に「**概要（定義）・標準プロパティ名・値仕様（型・桁数・利用可能文字・値域）**」を一元管理する。元は構想書の一節だったが独立文書に切り出した。情報定義・機能仕様・API仕様・品質保証 が参照する。参考実物なし・第一原理。

---

## 位置づけ（整合性モデル）

- 用語集は**全ドキュメントの語彙の唯一の源**。各業務用語の意味・**標準プロパティ名**・**値仕様**をここで一度だけ定義する。
- **情報定義は用語集の用語を「合成」してエンティティを作る**（用語集＝原子、情報定義＝合成）。属性の値仕様（型・桁数・文字種・値域）は用語集を源とし、情報定義は参照する。
- 整合性チェーン上は、構想と並ぶ**上流の共通基盤**。ここで語彙を揃えると、下流（情報定義→機能仕様→API仕様→実装→品質保証）の表記ゆれ・値仕様のブレが消える。

## 全体構成（章立てテンプレート）

| 章 | 内容 |
|----|------|
| 1. 概要 | 目的・参照関係 |
| 2. 用語定義 | 業務用語の辞書（概要・標準プロパティ名・値仕様） |
| 3. コード・区分値定義 | 列挙値の一覧（ステータス・区分など） |
| 4. 業務パラメータ | 区分・条件で変わる業務上の設定値（値・管理場所・変更手段） |
| 5. 略語・別名 | 略語／別名／表記ゆれの統一 |

> 以下、各章の書き方とコピペ用スケルトンを章番号に沿って示す。改訂履歴は持たず、要件トレーサビリティ文書で一元管理する。

---

## 1. 概要

目的（語彙の唯一の源）と、参照するドキュメント（情報定義・機能仕様・API仕様・品質保証 が本書を参照）を明記。

```markdown
## 1. 概要
- 本書は {システム名} の用語・標準プロパティ名・値仕様を一元定義する語彙の源。
- 各ドキュメント（情報定義・機能仕様・API仕様・品質保証）は本書を参照し、再定義しない。
```

## 2. 用語定義（辞書）

業務用語ごとに、概要・**標準プロパティ名**・**値仕様**を定める。情報定義の属性・APIの項目名は標準プロパティ名に従う。

| 用語 | 概要（定義） | 標準プロパティ名 | 型 | 桁数/長さ | 利用可能文字 | 値域 | 備考 |
|------|------------|----------------|----|----------|------------|------|------|
| 会員区分 | 利用者の区分 | memberType | 文字列(列挙) | - | - | 一般/学生/シニア（→ 3章） | コード値は3章 |
| 電話番号 | 連絡先電話 | phoneNumber | 文字列 | 最大15 | 半角数字・ハイフン | - | 任意 |
| 金額 | 税込金額 | amount | 整数 | - | 半角数字 | 0以上 | 円・税込・丸めは実装標準 |

## 3. コード・区分値定義

列挙型（ステータス・区分など）の**値の一覧**を、コード・表示名・説明で定める。

```markdown
### {区分名}（プロパティ：{プロパティ名}）
| コード | 表示名 | 説明 |
| ------ | ------ | ---- |
| ACTIVE | 有効 | 通常状態 |
| CANCELLED | キャンセル | 論理削除扱い |
```

## 4. 業務パラメータ

区分や条件によって値が変わる**業務上の設定値**（例：会員区分ごとの上限冊数・貸出期間）を一元管理する。
「値がいくつか」だけでなく、**どこで管理し・どう変えるか・変えたら何に効くか**まで書く（ハードコードの闇を作らない）。

```markdown
| パラメータ | 標準名 | 値（区分ごと） | 管理場所 | 変更手段 | 変更時の反映先 |
| ---------- | ------ | -------------- | -------- | -------- | -------------- |
| 上限冊数 | loanLimit | 一般=5 / 学生=10 | マスタ（会員区分） | 管理画面 | 機能仕様の上限チェック・品質保証の境界値 |
| 貸出期間 | loanPeriodDays | 一般=14 / 学生=28 | マスタ（会員区分） | 管理画面 | 返却期限の計算・品質保証の期限ケース |
```

- 管理場所は「マスタ（テーブル）／設定ファイル／ハードコード」のどれかを明記。ハードコードなら**変更にリリースが要る**ことを備考に書く。

## 5. 略語・別名

表記ゆれを防ぐため、略語・別名と正式名称の対応を残す。

```markdown
| 略語/別名 | 正式（標準）名称 | 備考 |
| --------- | ---------------- | ---- |
| 予約 | 鵜飼船予約情報 | 文書では正式名称を使う |
```

## 記載ルール（章横断の作法）

1. **語彙はすべてここで一度だけ定義し、各ドキュメントは参照する**（二重定義しない）。
2. **標準プロパティ名を必ず定める** — 情報定義の属性・APIの項目名・実装の変数名はこれに従い、一貫させる。
3. **値仕様（型・桁数・利用可能文字・値域）を明記** — 情報定義の値域・機能仕様の入力チェック・APIのバリデーションはこれを源として参照する。
4. **列挙値はコード・区分値定義（3章）で一覧化**（ステータス・区分など）。
5. **業務パラメータ（区分別の設定値）は4章に一元管理**し、値だけでなく管理場所・変更手段・反映先まで書く。
6. **略語・別名を残し、表記ゆれを防ぐ**（文書では正式名称を使う）。
7. **改訂履歴は持たず、要件トレーサビリティ文書で一元管理する**。

## 貫通線（このフォーマット群との関係）

- **値域の貫通線の"源流"**：用語集（値仕様・標準プロパティ名）→ 情報定義の値域 → 機能仕様の入力チェック → API仕様のバリデーション → 品質保証のケース。源がさらに一段上がる。
- **標準プロパティ名の一貫**：用語集 → 情報定義の属性 → APIの項目 → 実装の変数まで同じ名前で貫く。
- **業務パラメータ（4章）も値域の貫通線に乗る**：区分別の値（上限・期間など）→ 情報定義の業務ルール → 機能仕様のチェック → 品質保証の境界値ケース。値の変更は4章「変更時の反映先」を辿って追従する。
- **構想との分担**：構想は「向き・価値・なぜ」、用語集は「言葉の定義・値仕様」。構想から切り出して独立させた。

### {区分名}（プロパティ：{プロパティ名}）
