# 機能仕様 記載フォーマット（v0.5 たたき台）

**日付：** 2026-06-16（v0.2 実物ベース化 ／ v0.3 スケルトン整備 ／ v0.4 状態遷移・画面遷移図・外部連携を追加 ／ v0.5 全体構成の章番号と本文を一致させ、章ごとにフォーマットを記載）
**記録者：** 羽床 ＋ Claude

> [補足] 実在の予約系基幹システムの機能仕様書（約2500行）を元に、章立て・表構造・記載作法を抽出して汎用テンプレート化したもの。情報定義フォーマット（05）と対で使う。

---

## 位置づけ（整合性モデル）

- **情報定義 →【機能仕様】→ API仕様 → 実装** の縦チェーンの2段目。情報定義（源流）のエンティティ・値域・業務ルールを「各機能がどう操作するか」に具体化する。
- **すべての機能・画面項目・一覧の列が、情報定義の項番を参照する**（孤立を作らない）。
- 各機能・画面に**画面ID／機能ID**を振り、**要件 → 機能 → API → 品質保証**のトレーサビリティの起点にする。

## 1. 概要

システム概要・**情報定義への参照宣言**・環境差（表示名／テーマ等）を書く。

```markdown
## 1. 概要
- 本書は {システム名} の機能仕様。情報定義（info_def）の項番と連携する。
- 環境差：dev / prod の表示名・テーマ 等
```

## 2. 画面一覧

カテゴリで束ね、共通3列で書く。**全画面に画面ID**（命名規則：大文字スネーク。例 `DAILY_RECEPTION`）を振る＝ヘルプ・トレーサビリティ・品質保証対象IDの起点。「未実装」「暫定で非表示」も明記。

```markdown
### カテゴリ: {カテゴリ名}
| 画面名 | 概要 | 関連情報定義 |
| ------ | ---- | ------------ |
```

## 3. 画面遷移図

システム全体の画面回遊を1枚で俯瞰。各遷移に**トリガー（どのボタン/リンクで遷移するか）**を添える（Mermaid 等）。

```markdown
## 3. 画面遷移図
- 起点：ログイン → {初期画面}
（Mermaid flowchart 例）
[当日予約表] --予約追加--> (予約追加ダイアログ)
[当日予約表] --明細リンク--> [全体台帳]
[サイドメニュー] --> [各マスタ管理]
```

## 4. 共通コンポーネント一覧

画面一覧と同形式（名称｜概要｜関連情報定義）。サイドメニュー・ヘルプアイコン等、全画面共通のUI部品を列挙。

## 5. バッチ処理一覧・ジョブフロー（任意）

バッチを一覧化し、**実行スケジュール（契機）・依存関係**を明記する。さらに**ジョブフロー図**（Mermaid）で実行順序・依存を1枚で俯瞰する（画面遷移図のバッチ版）。**リラン・リカバリの全体方針は §15 実装標準**に集約する。

```markdown
### 5-1. バッチ一覧
| バッチ名 | 概要 | スケジュール（契機） | 依存（先行ジョブ） | 関連情報定義 |
| -------- | ---- | -------------------- | ------------------ | ------------ |

### 5-2. ジョブフロー図（実行順序・依存）
（Mermaid flowchart：ノード=ジョブ／エッジ=依存・実行順／ラベル=契機）
[日次集計] --完了後--> [請求データ作成] --> [請求書出力]
[月次締め] --毎月1日--> [支払情報作成]
```

## 6. 外部連携一覧（任意）

```markdown
| 連携名 | 相手 / 方向 | 概要 | 関連情報定義 |
```

## 7. 画面詳細仕様

**冒頭に「7-0. 各画面の共通仕様」**（ヘルプ・認証ガード等）を1回だけ書き、**適用対象**と**例外**を明示（情報定義の「共通項目外出し」と同じ思想）。各画面は次の標準セクションで書く。

| セクション | 書き方 |
|-----------|--------|
| 概要 / 関連情報定義 | 1文＋情報定義の項番 |
| 検索エリア | 項目｜入力形式｜必須｜**デフォルト値**｜説明 |
| 一覧表（列項目） | 列名｜説明｜**関連情報定義**（列ごとに項番を引く） |
| デフォルト並び順 | 明記。**最終ソートキーは作成日時降順**を共通規約に |
| 機能・ボタン | 機能｜配置｜フロー｜条件｜関連情報定義｜**API** |
| バリデーション | 項目｜チェック内容（**情報定義の業務ルール番号を参照**） |
| エラーメッセージ | 条件｜メッセージ（→ 品質保証の異常系ケースと1:1） |
| 削除制約 | 「参照されていれば削除不可」＝ 情報定義の FK/RESTRICT の画面表現 |

```markdown
#### {画面名}（画面ID: {SCREEN_ID}）
- 概要 / 関連情報定義：1-N, 3-N
- 検索エリア：| 項目 | 入力形式 | 必須 | デフォルト | 説明 |
- 一覧表（列項目）：| 列名 | 説明 | 関連情報定義 |
- デフォルト並び順：... → 作成日時（降順）
- 機能/ボタン：機能｜配置｜フロー｜条件｜関連情報定義｜API
- バリデーション：項目｜チェック内容（情報定義 3-x 参照）
- エラーメッセージ：条件｜メッセージ
- 削除制約：
```

## 8. 共通コンポーネント詳細仕様

サイドメニュー等の詳細＋**ダイアログ（追加/編集・確認）**。ダイアログは入力項目・モード切替・バリデーション・保存失敗時の挙動・操作を書く。

```markdown
#### {ダイアログ名}（画面ID: {DIALOG_ID}）
- 概要 / 関連情報定義：1-N, 3-N
- 入力項目：| 項目名 | 必須 | 入力形式 | 説明 |
- （モード切替あれば）モードA / モードB ごとの入力仕様
- バリデーション：項目｜チェック内容（情報定義 3-x 参照）
- 保存失敗時の挙動：／ 操作：ボタン → 呼ぶAPI → 結果
```

## 9. 画面共通仕様

表共通機能（ソート／フィルタ／EXCEL出力の共通方針）＋**帳票（EXCEL/PDF 出力）**。帳票は出力対象・ファイル名・列定義・並び順・スタイル/印刷設定を書く。

```markdown
#### {帳票名}（{SCREEN_ID}_EXPORT）
- 出力対象 / 件数制限：／ ファイル名：{名称}_{条件}.xlsx ／ シート構成：
- 列定義：| 列 | 列名 | 型 | 説明 | 表示形式 | 備考 |
- 並び順：画面の表示順と同一 ／ スタイル / 印刷設定 / 実装方法：
```

## 11. 外部連携詳細仕様（任意）

会計・EDI・ファイル連携など外部システムとの境界。**項目マッピング（相手項目 ↔ 情報定義の属性）**と**エラー/再送/冪等性**を必ず書く。

```markdown
#### {連携名}（{IF_ID}）
- 相手 / 方向：外部システムX / 送信（or 受信・双方向）
- 方式 / 契機：REST（or ファイルCSV/固定長・MQ）/ リアルタイム（or バッチ・イベント）
- 項目マッピング：| 相手項目 | 自システム（情報定義 1-N.属性） | 変換ルール |
- エラー処理 / 再送 / 冪等性：／ 認証 / セキュリティ：
```

## 12. 状態遷移

基幹システムは状態を持つエンティティが多い（予約ステータス・支払の算出ステータス・在庫状態…）。**最もバグが出やすい箇所**なので、各画面に散らさず、**状態を持つエンティティごとに1枚へ集約**する。状態一覧は情報定義の値域をそのまま使う。

```markdown
### 状態遷移：{エンティティ名}.{状態属性}
- 状態一覧（情報定義の値域）：A / B / C
- 遷移表：
  | From | To | トリガー（機能/API） | ガード条件 | 副作用 |
  | ---- | -- | -------------------- | ---------- | ------ |
  | 有効 | キャンセル | キャンセル確認ダイアログ / cancelReservation | 期日チェック | キャンセル料を記録 |
  | キャンセル | 有効 | 復活ボタン / updateReservationStatus | ADMIN権限 | キャンセル情報は保持 |
```

## 15. 実装標準（横断的な実装規約）

全機能・全実装が従う**横断的な規約を1か所に集約**する（＝横断的関心事の家）。各機能・各APIでは個別に方針を書かず、**本章を参照**する。値域・整合性は情報定義、環境・ツールはシステムアーキテクチャと**参照で繋ぎ、二重定義しない**。

| 標準 | 内容（テンプレ：プロジェクトで具体化） | 主な参照先 |
| ---- | -------------------------------------- | ---------- |
| 命名規約 | 変数・関数・ファイル・DBカラム・APIパス・ID の命名規則 | - |
| エラーハンドリング | 例外を握る層／伝播／ユーザー通知（トースト等）／リトライ・タイムアウト | API仕様（エラーコード） |
| ロギング・監査 | ログレベル・出力内容・**PII の扱い**・監査ログ・保持期間・出力先 | - |
| バリデーション実装 | どの層で検証するか／共通バリデータ（再定義しない） | 情報定義 §3・§5 |
| 日時・通貨・単位 | 日時形式・**タイムゾーン**／金額（税込・丸め）／数量・単位 | 情報定義（共通項目） |
| 国際化（i18n） | メッセージカタログ・対応言語・ロケール（多言語要件がある場合） | - |
| トランザクション・整合性 | トランザクション境界・楽観ロックの実装方針 | 情報定義 §4・API仕様 |
| セキュリティ実装 | 認証認可の実装・入力サニタイズ・機密情報の扱い・脆弱性対策 | システムアーキ §6 |
| 設定・環境差 | 設定値・フィーチャーフラグの扱い | システムアーキ §4 |
| コードスタイル・静的解析 | フォーマッタ・Lint・品質ゲート | システムアーキ §5 |
| バッチ・ジョブ運用 | **リラン・リカバリ方針**・冪等性・失敗時の通知・依存ジョブの扱い | 機能仕様 §5 |

> 各項目は「プロジェクトで具体化する」テンプレ。**横断的関心事はここが唯一の置き場**とし、機能ごとの重複記述を防ぐ。

## 記載ルール（章横断の作法）

1. **関連情報定義を「すべての粒度」で引く** — 画面・一覧の列・ダイアログ・バッチ・帳票・連携。縦リンクを徹底し孤立項目を作らない。
2. **バリデーション・計算ロジックは情報定義の業務ルール番号を参照**（再記述しない）。例：「定員チェック（情報定義 3-1-3）」「金額＝情報定義 3-2 に基づく算出」。
3. **全画面に画面ID（命名規則）を振る** — ヘルプ・トレーサビリティ・品質保証対象IDの起点。
4. **共通仕様を先に括り出す**（各画面共通・表共通・入力共通）。
5. **デフォルト並び順を必ず明記** — 最終ソートキー＝作成日時降順を共通規約に。
6. **削除制約を一覧に明記** — 「参照されていれば削除不可」は情報定義の FK/RESTRICT の画面側表現。
7. **エラーメッセージを（条件｜メッセージ）表で列挙** — そのまま品質保証の異常系ケースになる。
8. **「未実装・暫定・対象外」を明示** — 情報定義にあるが未実装、等のギャップを正直に書く。
9. **状態を持つエンティティは「状態遷移（12章）」に集約**（各画面に散らさない）。
10. **外部連携は項目マッピングとエラー/再送/冪等性を必ず書く**。
11. **改訂履歴は持たず、要件トレーサビリティ文書で一元管理する**／**API一覧に「利用画面」を必ず書く**。

## 貫通線（このフォーマット群を縦に貫くもの）

- **値域の貫通線**：情報定義の値域 →（機能仕様）画面の入力チェック・エラーメッセージ → API仕様のリクエストバリデーション → 品質保証のバリデーションケース。
- **状態の貫通線**：情報定義の値域（状態）→（機能仕様 12章）状態遷移表 → 品質保証の状態遷移ケース（不正遷移の拒否）。
- **トレーサビリティID連鎖**：要件ID → **画面ID／機能ID**（本書）→ API（13章の利用画面）→ 品質保証ID。
- **計算・整合性の参照**：金額・定員・状態遷移などは情報定義§3を参照し、機能仕様では「どの画面でいつ効くか」だけを書く（二重定義を避ける）。
