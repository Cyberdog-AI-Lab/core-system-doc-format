# 要件トレーサビリティ 記載フォーマット（v0.1 たたき台）

**日付：** 2026-06-18
**記録者：** 羽床 ＋ Claude

> [補足] 各ドキュメントに分散していた**改訂履歴を廃し、要件駆動で一元管理する単一文書**。要件チケットを軸に「いつ・どのドキュメントに・どの箇所を・どう反映したか」の時系列を1か所で追えるようにする。参考実物なし・第一原理。特定プラットフォーム手段には依存させない。

---

## 位置づけ（整合性モデル）

- 整合性モデルの**横断ビュー**。要件チケットを軸に、各ドキュメント（情報定義／機能仕様／API仕様／実装／品質保証）への**反映の時系列（＝改訂履歴）を一元管理**する。
- **各成果物ドキュメントからは個別の改訂履歴を廃止**し、本書に集約。すべての改訂を「**駆動した要件**」に紐づけて追えるようにする。
- 要件チケットの「反映状況テーブル」が**現時点のスナップショット**なのに対し、本書は**変更の時系列ログ**。現況（要件側）＋履歴（本書）で完全な追跡になる。

## 全体構成（章立てテンプレート）

| 章 | 内容 |
|----|------|
| 1. 概要 | 目的・対象ドキュメント |
| 2. トレーサビリティ・マトリクス | 要件 × 各ドキュメントの反映状態（現況の一覧） |
| 3. 要件別 反映タイムライン | 要件ごとの時系列改訂（いつ／対象／箇所／変更内容） |
| 4. 要件外の改訂ログ | 非要件駆動の改訂（構想改訂・リファクタ・誤記修正 等） |

> 以下、各章の書き方とコピペ用スケルトンを章番号に沿って示す。

---

## 1. 概要

目的＋対象ドキュメント宣言（要件／情報定義／機能仕様／API仕様／品質保証）。「各ドキュメントの改訂履歴を本書に集約し、要件IDを軸に時系列で追う」ことを明記。

```markdown
## 1. 概要
- 本書は {システム名} の改訂履歴を、要件IDを軸に一元管理する。
- 対象ドキュメント：要件 / 情報定義 / 機能仕様 / API仕様 / 実装 / 品質保証
- 各成果物ドキュメントは個別の改訂履歴を持たない（本書に集約）。
```

## 2. トレーサビリティ・マトリクス

要件 × ドキュメントのグリッドで**現況**を一覧する。**この表がダッシュボードの進捗タイムラインの読み取り元**（＝書き方を揃えるほど自動で見える化される）。

```markdown
| 要件ID | タイトル | 情報定義 | 機能仕様 | API仕様 | アーキテクチャ | 実装 | 品質保証 |
| ------ | -------- | -------- | -------- | ------- | -------------- | ---- | -------- |
| R-001  | 蔵書を検索 | 済 2026-06-10 1-1 | 済 2026-06-12 F-001 | 済 2026-06-14 API-001 | − | 済 2026-06-18 PR#12 | 済 2026-06-20 QA-007 |
| R-002  | 貸出を登録 | 済 2026-06-10 1-4 | 反映中 | 未 | − | 未 | 未 |
| NFR-01 | 照会1秒以内 | − | − | − | 済 2026-06-11 ADR-001 | 済 2026-06-19 PR#15 | 未 |
```

**セルの書き方（これだけ覚えればよい）**

- セル＝「**状態 日付 反映先ID**」を半角スペース区切りで書く。例：`済 2026-06-20 QA-007`
- 状態は3語だけ：`未` ／ `反映中` ／ `済`（要件チケットの反映状況と同じ言葉）
- **対象外の層は `−`**（例：機能要件のアーキテクチャ列、非機能要件の情報定義〜API列）
- 日付＝その状態にした日。反映先ID＝情報定義 1-N／F-xxx／API-xxx／ADR-xxx／PR番号／QA-xxx
- 未の場合は `未` だけでよい（日付・IDは不要）

チケットの代表状態（「実装中」など）は**書かない**。どこまで済んだかは表から自動で分かる（ダッシュボードが表示する）。手で書くと表とズレていくため。

## 3. 要件別 反映タイムライン

本書の心臓。**要件ごとに**、反映の**時系列**を積む。1反映＝1行（＝旧・各ドキュメントの改訂1件）。

```markdown
### R-{番号} {タイトル}
| 日付 | 対象ドキュメント | 箇所 | 変更内容 | 版/コミット | 実施者 |
| ---- | ---------------- | ---- | -------- | ----------- | ------ |
| 2026-06-18 | 情報定義 | 1-5 | エンティティ「貸出」を追加 | - | 羽床 |
| 2026-06-19 | 機能仕様 | F-003 | 貸出画面に定員チェックを追加 | - | 羽床 |
| 2026-06-20 | API仕様 | API-007 | createLoan に楽観ロックを追加 | - | 羽床 |
| 2026-06-21 | 品質保証 | QA-012 | 定員超過時エラーを検証（合格） | - | 羽床 |
```

- **箇所は成果物の番号で指す**（情報定義 1-N／機能仕様 F-xxx／API仕様 API-xxx／品質保証 QA-xxx）。
- 取り下げ・差し戻しも時系列で残す（消さない）。要件の取り下げコンテキストと整合。

## 4. 要件外の改訂ログ（非要件駆動）

要件に紐づかない改訂（誤記修正・リファクタ・構想改訂など）を、ここにまとめて時系列で残す。

```markdown
| 日付 | 対象ドキュメント | 箇所 | 変更内容 | 種別 |
| ---- | ---------------- | ---- | -------- | ---- |
| 2026-06-18 | 情報定義 | 全体 | 用語を用語集に合わせて統一 | リファクタ |
```

- 種別：`誤記修正` / `リファクタ` / `構想改訂` / `その他`。

---

## 記載ルール（章横断の作法）

1. **すべての改訂を「日付・対象ドキュメント・箇所・変更内容」で記録する**。
2. **要件駆動の改訂は必ず要件IDに紐づける**（要件別タイムラインへ）。
3. **非要件駆動（誤記・リファクタ・構想改訂）は「要件外の改訂ログ」へ**。
4. **現況はマトリクス、履歴はタイムラインで**（役割分担。二重管理しない）。
5. **各成果物ドキュメントは個別の改訂履歴を持たない**（本書に集約）。
6. **箇所は成果物の番号で指す**（情報定義 1-N／機能仕様 F-xxx／API仕様 API-xxx／品質保証 QA-xxx）。
7. **取り下げ・差し戻しも時系列で残す**（消さず蓄積）。
8. **マトリクスのセルは「状態 日付 反映先ID」の順・状態は「未／反映中／済」・対象外は「−」で統一**（§2）。ダッシュボードがこの表を読むため、書式を崩すと進捗が見えなくなる。

## 貫通線（このフォーマット群との関係）

- **トレーサビリティの集約点**：要件ID・情報定義 1-N・機能 F-xxx・API API-xxx・品質保証 QA-xxx が、本書で時系列に交差する。
- **現況＋履歴**：要件チケットの**反映状況テーブル＝現況**、本書の**タイムライン＝履歴**。両者で「今どこまで」「いつどう変わったか」を完全に追える。
- **改訂履歴の単一の置き場**：他フォーマットは個別の改訂履歴章を持たず、本書を唯一の改訂履歴とする。
