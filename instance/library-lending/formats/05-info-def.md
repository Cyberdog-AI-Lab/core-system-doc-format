## 位置づけ（整合性モデル）

- **情報定義は全仕様の源流**。機能仕様・API仕様・実装はすべてここから派生する（→ 情報定義 → 機能仕様 → API仕様 → 実装）。
- 1ファイルに **論理層**（エンティティ・リレーション・業務ルール）を集約する。物理設計は同居させてもよいが、**章で明確に分ける**。
- 「読めば、このシステムが扱う情報のすべてと、その正しさのルールが分かる」状態をゴールにする。

## 1. エンティティ定義

### 1-1. 共通項目を最初に1回だけ定義する

全エンティティに共通する項目（作成日時・更新日時など）は、**冒頭で1回定義**し、各エンティティ表では省略する。重複記載を避け、ブレを防ぐ。

```markdown
### 1-0) 共通項目（全エンティティ共通）
| 属性名 | データ型 | 必須 | 説明 | メンテルール |
| :----- | :------- | :--- | :--- | :----------- |
| 作成日時 (createdAt) | 日時(ISO8601) | 必須 | レコード作成日時 | 新規作成時に自動設定。更新時は変更しない。 |
| 更新日時 (updatedAt) | 日時(ISO8601) | 必須 | レコード更新日時 | 新規作成時に自動設定。更新時に上書き。 |
```

### 1-2. 各エンティティのスケルトン（コピペ用）

```markdown
### 1-N) {エンティティ名}
- **概要**：{1〜2文。何を表す情報か、親子関係や役割}
- **主キー**：{主キー属性}
- **業務キー（ユニーク）**：{業務上一意になる属性の組}　← あれば
- **一意制約**：{（属性A, 属性B）で一意 など}　← あれば

| 属性名 | データ型 | 必須 | 説明 | 制約 / 備考 | 関連先 |
| ------ | -------- | ---- | ---- | ----------- | ------ |
| {属性} | {型} | 必須/任意 | {意味} | {値域・初期値・計算式など} | {参照先エンティティ or -} |

> {補足ノート：値域の詳細、初期データ、計算式、運用ルール、変更履歴など}
```

### 1-3. 属性テーブルの列定義（6列で固定）

| 列 | 役割 | 書き方の原則 |
|----|------|-------------|
| 属性名 | 論理名（日本語） | 必要なら `(camelCase)` で物理名も併記 |
| データ型 | 整数/UUID・文字列・日時・数値・真偽値 等 | 抽象型でよい（DB型は物理設計章で） |
| 必須 | 必須 / 任意 | NULL可否を明示 |
| 説明 | その属性が何か | 1行で。単位・税込/税抜などもここ |
| 制約 / 備考 | 値域・初期値・計算式・FK条件 | **「値域：『A』『B』」と列挙する** |
| 関連先 | 参照先エンティティ名 | FKならエンティティ名、なければ `-` |

> 属性の**標準プロパティ名・値仕様（型・桁数・利用可能文字）は別文書「用語集」を源**とし、本書はその用語を組み合わせてエンティティを構成する（用語集＝原子、情報定義＝合成）。エンティティ固有の値域はここで明示する。

## 2. リレーション定義（総覧）

エンティティ間の関連を**一覧で総覧**する。記法を固定する。

```markdown
- {エンティティA} **N — 1** {エンティティB}（FK: {属性名}）
- {エンティティA} **1 — N** {エンティティC}（FK: {属性名}）
- {エンティティA} **N — 0..1** {エンティティD}（FK: {属性名}、NULL可）

**主なユニーク制約**
- {エンティティ}：（属性A, 属性B）ユニーク
```

- カーディナリティは `1 — N` / `N — 1` / `N — 0..1`（任意参照）で表現。
- 末尾に**ユニーク制約だけを集めた小節**を置き、実装の制約に直結させる。

## 3. 代表的な業務ルール

### 3-1. カテゴリで束ねる

「情報目線の業務ルール」をカテゴリで束ねる。

| カテゴリ | 例 |
|----------|----|
| 登録・割当 | 一意性の担保、定員チェック、有効期間チェック |
| **バリデーション（データ整合性担保）** | 単項目・項目間・エンティティ間の妥当性チェック（→ 3-2） |
| 金額計算 | 料金の算出式、税の扱い、内訳の定義 |
| 変更・キャンセル | 変更可否マトリクス、料率の段階、履歴記録先 |
| 運用・整合性 | 合計の一致（親 ＝ ∑子）、削除ポリシー（論理削除/RESTRICT） |
| 集計・支払 | 集計単位、対象の絞り込み、明細との突合 |
| アルゴリズム | 自動割当などの手順・優先順位・出力 |

書き方：**擬似コード**（料率の IF 分岐）／**判定表**（前日・当日の変更可否）／**境界例**（利用日12/1なら11/30は前日…）を積極的に使う。

### 3-2. バリデーション（業務データの整合性担保ルール）

「業務データが正しい状態であるための条件」を **3層**に分けて漏れなく書く。第1章の属性表に書いた**必須・値域・関連先（FK）・一意制約**を、ここで *いつ・どう検証し・違反したらどうするか* として手続き化する。

| 層 | 何を検証するか | 例 |
|----|----------------|----|
| **単項目** | 1属性の妥当性 | 必須／型／桁・長さ／フォーマット（メール・電話・ISBN）／値域（列挙値・0以上） |
| **項目間**（同一レコード） | 複数属性の関係 | 適用開始 ≤ 適用終了、貸出日 ≤ 返却期限、「弁当代込みフラグ=True なら一律弁当料金は必須」 |
| **エンティティ間**（レコード横断） | 横断的な整合 | 参照先の存在、業務キーの一意、合計一致（親.人員 ＝ ∑子.人数）、状態遷移の妥当性、定員超過なし |

各ルールに、最低限この3点を必ず添える：

- **検証タイミング**：登録時 ／ 更新時 ／ 状態遷移時 ／ バッチ算出時
- **違反時の挙動**：エラーで拒否 ／ 警告のみ ／ 自動補正（既定値・丸め）
- **担保場所**：DB制約（UNIQUE / FK / CHECK / NOT NULL）か、アプリ業務ルールか（→ §4「整合性の担保場所」の2分類。**NoSQL では (a) 宣言的整合性もここに書き起こす**）

**記載スケルトン（コピペ用）**

```markdown
#### V-{番号} {ルール名}
- 対象：{エンティティ.属性 もしくは エンティティ間}
- 層：単項目 / 項目間 / エンティティ間
- 条件：{満たすべき条件。擬似コード可}
- タイミング：{登録時 / 更新時 / 状態遷移時 / バッチ時}
- 違反時：{拒否 / 警告 / 補正}
- 担保場所：{DB制約名 もしくは アプリ業務ルール}
```

> [補足] バリデーションは第1章の「値域・必須・関連先」と表裏一体。属性表が *静的な定義*、3-2 が *動的な検証手続き*。両者を **対象（エンティティ.属性）でひも付ける**と、定義したのに検証していない／検証しているのに定義にない、という抜けが見つけやすい。

## 4. 物理データ設計・実装方針（任意）

### 4-1. 大原則 — 論理は共通、物理で分岐する

§1〜3（エンティティ・リレーション・業務ルール）は **DB非依存**。RDBでもNoSQLでも同じものを書く。分かれるのは§4〜5（物理設計）で、**キー設計の書き方がDBの特性で根本から変わる**。設計の出発点からして逆である。

| 観点 | RDB（リレーショナル） | NoSQL（KVS/ドキュメント型） |
|------|----------------------|------------------------------------------|
| 設計の起点 | **データ構造**（正規化された形を先に決める） | **アクセスパターン**（先に全クエリを洗い出す） |
| 結合 | JOIN で都度結合 | JOINなし → **非正規化・複製**で対応 |
| 整合性の担保 | **DBが保証**（FK・UNIQUE・CHECK・トランザクション） | **アプリ＋DB機能**（条件付き書込・トランザクション）で最小限 |
| スキーマ | 固定（事前定義） | 柔軟（スキーマレス） |
| キーの主役 | 主キー ＋ 外部キー ＋ インデックス | パーティションキー/ソートキー ＋ セカンダリインデックス |
| 一意性の表現 | UNIQUE 制約 | キー設計 ＋ 条件付き書込で表現 |

> 出発点が逆なのが核心：RDBは「正しい構造」から入ってクエリは後付け、NoSQLは「必要なクエリ」から入ってキーを決める。

### 4-2. 整合性をどこで担保するか（業務ルール章の厚みが変わる）

整合性ルールを**2種類に分けて**考えると、DB差がどこに出るかが見える。

| 整合性の種類 | 例 | RDB | NoSQL |
|--------------|----|-----|-------|
| **(a) DBが宣言的に守れるもの**（一意性・参照整合性・値域） | 「1蔵書＝同時1貸出」「FK参照」「状態は列挙値のみ」 | UNIQUE/部分UNIQUE・FK・CHECK の**1行で保証** | 宣言的制約がない → **条件付き書込＋トランザクションをアプリ（業務ルール章）に明記** |
| **(b) 元々アプリ集計が必要なもの**（合計・上限・件数） | 「現在貸出冊数 ＜ 上限」「親.人員 ＝ ∑子」 | 集約クエリ＋アプリ判定 | Query-then-count＋アプリ判定 |

- **(a) は NoSQL で業務ルール章へ"移動"して厚くなる**。RDBが制約名1行で済ませていたものを、手続きとして書き起こす必要があるため。
- **(b) は DB 種別で差が出にくい**（どちらもアプリで集計・判定する）。

> **精緻化（2026-06-15、図書貸出ドメインの検証より）**：当初 v0.2 では「NoSQLだと業務ルール章が一律に厚くなる」と書いたが、検証の結果 **「DBが宣言的に守れる整合性（a）ほど、NoSQLでは業務ルール章へ移動して厚くなる」** が正確。象徴例＝「二重貸出禁止」：RDBは部分ユニークインデックス1行／NoSQLはトランザクション＋条件付き書込。（図書貸出ドメインでフォーマットを実際に埋めて検証した結果による）。

> [補足] 元にした実プロジェクトは NoSQL 採用のため、業務ルール章と整合性担保（条件付き書き込み・楽観ロック・全件走査での参照チェック）が特に厚い。RDBなら(a)の整合性はFK・UNIQUE制約に寄せられる。

## 5. 物理エンティティ設計詳細（任意）

### 5-1. RDB 版（キー設計）

特性：正規化＋JOIN、整合性はDB制約で保証、スキーマ固定。記載する項目：

- **テーブル定義**：物理テーブル名・カラム物理型（`VARCHAR(n)` `INT` `TIMESTAMP` 等）
- **主キー**：サロゲート（採番）かナチュラル（業務キー）かを明記
- **外部キー制約**：参照先＋`ON DELETE` / `ON UPDATE` の挙動（`RESTRICT` / `CASCADE` / `SET NULL`）
- **一意制約（UNIQUE）／CHECK／NOT NULL**：論理層の「業務キー・値域」をDB制約に落とす
- **インデックス設計**：検索・結合・ソートに使う列、**複合インデックスの列順**、カバリングの要否
- **正規化レベル**：基本は第3正規形。意図的に非正規化する場合は**理由を明記**

```markdown
#### 物理テーブル：{table_name}（論理：1-N {エンティティ名}）
| カラム | 型 | NULL | キー | 制約 / インデックス |
| ------ | -- | ---- | ---- | ------------------- |
| id | BIGINT | NOT NULL | PK | AUTO_INCREMENT |
| parent_id | BIGINT | NOT NULL | FK→parent(id) | ON DELETE RESTRICT, INDEX |
- UNIQUE: (col_a, col_b)
- INDEX: (col_x, col_y)  ※列順は検索条件に合わせる
```

### 5-2. NoSQL 版（キー設計）

特性：アクセスパターン駆動、JOINなし→非正規化、整合性はアプリ＋DB機能で担保、スキーマレス。記載する項目：

- **アクセスパターン一覧（最初に書く＝設計の起点）**：ユースケース → 取得方法 → キー条件
- **パーティションキー**：集約境界（一緒に読む単位）／負荷分散の単位
- **ソートキー**：`<EntityType>#<id>` プレフィックスでエンティティ混在を区別、範囲検索（前方一致）
- **セカンダリインデックス**：別の検索軸。**アクセスパターン分だけ最小限**に足す
- **非正規化・複製方針**：どの属性をどこへ複製し、更新時にどう同期するか
- **整合性担保**：楽観ロック（`version`）／条件付き書き込み（存在チェック等）／トランザクション境界（同一パーティション内）／冪等性
- **物理上限への対策**：ホットパーティション・アイテムサイズ上限（製品ごとに上限あり）

```markdown
#### アクセスパターン一覧
| ユースケース | 取得方法 | キー条件 |
| ------------ | -------- | -------- |
| 予約詳細取得 | 主キー取得 | PK = RESERVATION#<id> |
| ある日の予約一覧 | セカンダリインデックス | 索引キー = DATE#<日付> |

#### 項目設計（論理：1-N {エンティティ名}）
- パーティションキー: `...#<id>` / ソートキー: `<EntityType>#<id>`
- セカンダリインデックス: キー=`...` ソート=`...`（用途：...）
- 複製属性: {属性}（複製元：...、同期タイミング：...）
- 整合性: 新規=存在チェック付き書き込み / 更新=version楽観ロック
```

### 5-3. 論理と物理の対応づけ

- 論理エンティティ番号（1-N）と物理項目を必ず対応づける。
  - 例：「1-1) 予約情報 → RDB: `reservations` テーブル ／ NoSQL: PK=`RESERVATION#<id>`, SK=`METADATA`」

## 記載ルール（章横断の作法）

このフォーマットの本体。**なぜこう書くか**まで含めて12箇条にまとめる。

1. **主キーと業務キーを分けて書く** — 主キーは採番ID、業務上の一意性（例：親プラン×日付×船）は「業務キー／一意制約」として別記。実装の一意制約に直結する。
2. **値域は文字列で明示する** — 「値域：『貸切』『通常』」のように取りうる値を列挙。ステータス系は **固定値の一覧表**（ID・コード・表示順・説明）まで添える。
3. **派生・計算で出せる項目は保持しない** — 例：「金額項目は持たない（計算で算出）」。何を保持し何を計算するかの線引きを明記する。
4. **スナップショットが要る項目だけ、要る理由とともに保持** — 例：マージン率は支払時点の値を明細に複写。料金は「変更時に新レコード」とし、過去データの金額を常に再現可能にする。
5. **属性レベルの変更履歴を日付つきで残す** — 「算出ステータス（2026-01-27追加）」のように補足ノートに残す。**情報定義が"生きている"証拠**であり、後から経緯を辿れる。
6. **廃止は物理削除せず『廃止（日付）』で残す** — 番号も欠番にしない（例：`1-23) 入金情報（廃止）`）。消すと「なぜ無いのか」が分からなくなる。
7. **抽象定義に必ず具体サンプルを添える** — 「設定例」「初期データ」表を置く。理解が上がり、AI生成時の Few-shot 例にもなる。
8. **計算式は『＝』で明記する** — 「合計料金 ＝ 全体料金 ＋ ∑（個数 × 単価）」。曖昧さを消す。
9. **エンティティ定義と業務ルールを相互参照で繋ぐ** — 「業務ルール 3-2 参照」のようにクロスリンク。定義（静）とルール（動）が分断しない。
10. **業務ルールは擬似コード・判定表・具体例で書く** — IF 文の擬似コード、可否マトリクス、日付の境界例。文章だけにしない。
11. **論理と物理を章で分離する** — エンティティ・リレーション・業務ルール（論理）と、キー設計（物理）を混ぜない。源流性を保つ。
12. **共通の外出し＋6列固定で表記をブレさせない** — フォーマットが一定だと、人もAIも差分を追いやすく、生成・レビューが安定する。

## 貫通線（下流フォーマットへ貫くもの）

情報定義は **値域・状態・業務ルールの源流**。ここで定めた語彙が、機能仕様 → API仕様 → 品質保証へ一本で貫く。

- **値域**：情報定義の値域 → 機能仕様の入力チェック → API仕様のリクエストバリデーション → 品質保証のバリデーションケース。
- **状態**：情報定義の値域（状態）→ 機能仕様の状態遷移 → 品質保証の不正遷移ケース。
- **整合性の担保場所**（§4-2）：DBが守るか／アプリ（業務ルール）が守るか → API仕様の「トランザクション境界」で実体化。

## コピペ用・空スケルトン（文書全体・最小セット）
