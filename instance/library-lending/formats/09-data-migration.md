# データ移行 記載フォーマット（v0.1 たたき台）

**日付：** 2026-06-18
**記録者：** 羽床 ＋ Claude

> [補足] データ移行の設計・実施を扱う。**初回移行（旧システム → 新システム）に限らず、リリースに伴う移行（スキーマ変更・データ変換・バックフィル・データ修正）も含め、すべての移行を時系列で蓄積する「移行台帳」**。移行先は情報定義のエンティティ・属性に直結する。参考実物なし・第一原理。特定プラットフォーム手段には依存させない。

---

## 位置づけ（整合性モデル）

- データ移行は**情報定義（移行先のデータ構造）に直結**する。情報定義のスキーマ変更（＝改訂）には、しばしば移行が随伴する。
- **初回移行だけでなく、リリース随伴の移行も同じ場所に蓄積**する（1移行＝1記録）。リリース随伴移行は、駆動した**要件**と**リリース**に紐づく。
- 機能の流れ（情報定義 → 機能仕様 → API仕様）とは独立した**データの引っ越し**の軸。やりっぱなしにせず、検証・ロールバック・結果まで残す。

## 全体構成（章立てテンプレート）

| 章 | 内容 |
|----|------|
| 1. 概要 | 目的・対象 |
| 2. 移行方針 | 方式・停止可否・ロールバック・検証基準の共通方針 |
| 3. 移行記録 | 1移行のテンプレート（トリガー/対象/マッピング/手順/検証/ロールバック/結果） |
| 4. 移行一覧（総覧） | いつ・何を・結果 |

> 以下、各章の書き方とコピペ用スケルトンを章番号に沿って示す。改訂履歴は持たず、要件トレーサビリティ文書で一元管理する。

---

## 1. 概要

目的＋対象（移行元システム／対象データ）と、関連ドキュメント（情報定義＝移行先・要件＝移行要件）。

```markdown
## 1. 概要
- 本書は {システム名} のデータ移行（初回＋リリース随伴）を設計・記録する。
- 移行先は情報定義のエンティティ・属性。関連：要件（移行要件）／情報定義（移行先スキーマ）。
```

## 2. 移行方針（共通の考え方）

個々の移行に先立つ全体方針。

- **移行方式**：一括（ビッグバン）／段階（フェーズ分割）／並行稼働（新旧併走）
- **停止可否**：停止移行（メンテ時間）／無停止移行（オンライン）
- **リハーサル方針**：本番前に本番相当データで予行するか・どの環境で・何を確認するか（本番でぶっつけをしない）
- **ロールバック方針**：失敗時にいつまでに・どう戻すか（旧データ保全・切戻し）
- **検証基準の共通方針**：件数一致・整合性・サンプル突合・受け入れ条件
- **データクレンジング方針**：不正データ・重複・欠損の扱い

```markdown
## 2. 移行方針
- 方式：一括 / 段階 / 並行稼働
- 停止可否：停止 / 無停止
- リハーサル：{要否・実施環境・確認すること}
- ロールバック：{切戻し条件・手順の方針}
- 検証基準：件数一致 / 整合性 / サンプル突合 / 受け入れ
- クレンジング：{不正・重複・欠損の扱い}
```

---

## 3. 移行記録

1移行＝1記録。**初回も、リリース随伴も、データ修正も**、すべてここに時系列で積む。各記録を次の項目で書く。

| 項目 | 内容 |
|------|------|
| 移行ID | `MIG-xxx`（不変） |
| **トリガー** | 初回移行 ／ リリース `vX.Y` 随伴 ／ データ修正 ／ 障害復旧（＋対応要件ID） |
| 予定日 / 実施日 | |
| 目的・背景 | なぜこの移行が必要か |
| **対象** | 移行元（システム・テーブル）／対象エンティティ（**情報定義 1-N**）／範囲・件数（概算） |
| **データマッピング** | 移行先 ↔ 移行元 ↔ 変換ルール ↔ 必須 ↔ 欠損/デフォルト（下表） |
| 移行手順 | 事前準備 → 抽出 → 変換 → 投入 → 突合検証 → カットオーバー |
| **リハーサル** | 実施日・環境・結果（本番前の予行。やらない場合は理由を残す） |
| 検証・受け入れ基準 | 件数一致／整合性／サンプル突合／合否基準 |
| ロールバック手順 | 失敗時の切戻し |
| 停止/無停止・影響範囲 | 業務影響・ダウンタイム |
| **実施結果** | 実施日・結果（成功/失敗）・件数・残課題（消さず残す） |

**データマッピング表**（核）：

| 移行先（新・情報定義 1-N.属性） | 移行元（旧.列） | 変換ルール | 必須 | 欠損/デフォルト |
|--------------------------------|----------------|-----------|------|----------------|
| 利用者.会員区分ID | members.type | コード変換（"A"→"一般"） | 必須 | エラーで停止 |
| 蔵書.状態 | books.flag | フラグ→列挙値へ正規化 | 必須 | 既定値「貸出可」 |

**コピペ用スケルトン（移行1件）**

```markdown
### MIG-{番号} {タイトル}
- トリガー：初回 / リリース vX.Y / データ修正（対応要件：R-xxx）
- 予定日 / 実施日：
- 目的・背景：
- 対象：移行元 {...} ／ 対象エンティティ 1-N ／ 件数 約{...}
- データマッピング：
  | 移行先（1-N.属性） | 移行元 | 変換ルール | 必須 | 欠損/デフォルト |
- 移行手順：1.事前準備 2.抽出 3.変換 4.投入 5.突合検証 6.カットオーバー
- リハーサル：{実施日・環境・結果／やらない場合は理由}
- 検証・受け入れ基準：件数一致 / 整合性 / サンプル突合
- ロールバック手順：
- 停止/無停止・影響範囲：
- 実施結果：{実施日・成功/失敗・件数・残課題}
```

## 4. 移行一覧（総覧・任意）

移行を1行ずつ俯瞰し、「いつ・何を移行し・結果はどうだったか」を一望する。

```markdown
| 移行ID | トリガー | 実施日 | 対象エンティティ | 件数 | 結果 |
| ------ | -------- | ------ | ---------------- | ---- | ---- |
```

## 記載ルール（章横断の作法）

1. **すべての移行を1記録（時系列）で残す** — 初回・リリース随伴・データ修正・障害復旧、区別なく蓄積。
2. **移行先は情報定義のエンティティ・属性で指す**（1-N.属性）。曖昧にしない。
3. **データマッピングに変換ルール・必須・欠損/デフォルトを必ず明記**（変換の取りこぼしを防ぐ）。
4. **検証・受け入れ基準とロールバック手順を必ず書く**（やりっぱなしにしない）。本番前は**リハーサル**を行い結果を残す（やらない場合は理由を残す）。
5. **リリース随伴移行は、駆動した要件IDとリリースに紐づける**（トレーサビリティ）。
6. **実施後は結果（件数・残課題）を記録**（消さず蓄積）。
7. **改訂履歴は持たず、要件トレーサビリティ文書で一元管理する**。

## 貫通線（このフォーマット群との関係）

- **情報定義との連動**：移行先＝情報定義のエンティティ・属性。情報定義のスキーマ改訂には移行が随伴する。
- **要件・リリースとの連動**：リリース随伴移行は「要件 → 情報定義の改訂 → 移行」と連鎖し、要件トレーサビリティ文書に反映される。
- **品質保証との連動**：移行の検証（件数・整合性・突合）は品質保証の「整合性／データ」観点と対応づく。
