# 品質保証 記載フォーマット（v0.1 たたき台）

**日付：** 2026-06-18（旧「テスト」フォーマット v0.2 を「品質保証」として再定義）
**記録者：** 羽床 ＋ Claude

> [補足] 6分類の「テスト」を**品質保証**に改名・再定義。検証の**実施記録を蓄積する場所**として「いつ・何に対して・どのようなテストを・どういう観点で行い・何が保証できたか」を残し、各成果物ドキュメントの該当箇所に紐づける。参考実物なし・第一原理。特定プラットフォーム手段には依存させない。

---

## 位置づけ（整合性モデル）

- 品質保証は整合性モデルの**横の保証**。2系統で検証する：
  - **要件視点**：ユーザー要望が満たされているか（要件 → 品質保証）
  - **成果物視点**：仕様通り動くか・バグがないか（情報定義／機能仕様／API仕様／実装 → 品質保証）
- 役割は「**いつ・何に対して・どのようなテストを・どういう観点で行い・何が保証できたか**」を**蓄積する場所**。テストケースの台帳であると同時に、**保証の証跡**。
- 各記録は**成果物ドキュメントの該当箇所を指す**（要件 R-xxx／情報定義 1-N／機能仕様 F-xxx／API仕様 API-xxx）。要件チケットの「反映状況（品質保証列）」と相互参照し、**トレーサビリティ連鎖を閉じる**。

## 全体構成（章立てテンプレート）

| 章 | 内容 |
|----|------|
| 1. 概要 | 目的・関連ドキュメント |
| 2. 共通定義 | ID採番・テスト種別/レベル・観点・結果ステータス |
| 3. 品質保証記録 | 1記録の記載テンプレート（いつ/何に/どんな/観点/保証/箇所指示/結果） |
| 4. 網羅の考え方 | 2系統（要件視点／成果物視点）× カバレッジ |
| 5. 品質保証一覧（総覧） | 記録の一覧表（任意） |
> 以下、各章の書き方とコピペ用スケルトンを章番号に沿って示す。

## 1. 概要

目的＋関連ドキュメント宣言（要件／情報定義／機能仕様／API仕様への参照）。「検証の実施記録を蓄積し、各成果物の何が保証されているかを一望できる」ことを明記。

```markdown
## 1. 概要
- 本書は {システム名} の品質保証記録を蓄積し、何が・どこまで保証されているかを管理する。
- 関連ドキュメント：要件 / 情報定義 / 機能仕様 / API仕様
```

## 2. 共通定義

品質保証記録で使う語彙・採番ルールを冒頭に1回定義する。

### 2-1. ID採番
- 形式：`QA-{連番}`（不変）。

### 2-2. テスト種別・レベル
`単体` / `結合` / `受け入れ` / `非機能`（性能・セキュリティ等）/ `回帰` / `探索的`

### 2-3. 観点
`正常系` / `異常系` / `境界値` / `整合性` / `状態遷移` / `性能` / `セキュリティ` / `可用性` / `ユーザビリティ` / `互換性`
（非機能の観点は要件の「非機能種別タグ」と対応させる）

### 2-4. 結果ステータス
`合格` / `不合格` / `保留` / `未実施`

**スケルトン（共通定義の章）**

```markdown
## 2. 共通定義
### 2-1. ID採番：QA-{連番}（不変）
### 2-2. テスト種別・レベル：単体/結合/受け入れ/非機能/回帰/探索的
### 2-3. 観点：正常/異常/境界/整合性/状態遷移/性能/セキュリティ/可用性/ユーザビリティ/互換性
### 2-4. 結果ステータス：合格/不合格/保留/未実施
```

## 3. 品質保証記録

1検証＝1記録。検証するたびに**蓄積**していく。各記録を次の項目で書く。

| 項目 | 内容 |
|------|------|
| ID | `QA-xxx`（不変） |
| **いつ（実施日）** | 実施日／期間 |
| **何に対して（対象）** | 対象種別＋**成果物ドキュメントの箇所**（要件 R-xxx／機能仕様 F-xxx／API仕様 API-xxx／情報定義 1-N） |
| **どのようなテスト（種別・レベル）** | 2-2 から |
| **どういう観点（観点）** | 2-3 から（複数可） |
| テスト内容 | 前提・入力・操作・期待結果（サンプルは情報定義のサンプルを利用） |
| **何が保証できたか** | この検証で**言えるようになったこと**（どの受け入れ条件／仕様が満たされたか） |
| 結果 | `合格`/`不合格`/`保留` ＋ **エビデンス** |
| 不具合 | 不合格時：不具合ID・内容 |
| **環境** | どの環境で実施したか（開発／検証／本番相当 等。アーキ§4の環境名で書く） |
| 実施者 | |
| 由来 | どの仕様・値域・業務ルール・状態遷移から導いたケースか |

**「何が保証できたか」の書き方**：合否だけで終わらせず、「**何が言えるようになったか**」を成果物の箇所で示す。例：「R-012 の受け入れ条件①②が満たされることを確認」「機能仕様 F-003 の定員超過時にエラー表示されることを保証」「API仕様 API-003 の CONFLICT 応答を保証」。

**コピペ用スケルトン（記録1件）**

```markdown
### QA-{番号} {タイトル}
- いつ（実施日）：YYYY-MM-DD
- 何に対して（対象）：要件 R-xxx ／ 機能仕様 F-xxx ／ API仕様 API-xxx ／ 情報定義 1-N
- どのようなテスト（種別/レベル）：結合
- どういう観点：異常系 / 整合性
- テスト内容：前提・入力・操作・期待結果
- 何が保証できたか：{どの受け入れ条件/仕様が満たされたか（成果物の箇所つき）}
- 結果：合格 ／ エビデンス：
- 不具合：（不合格時のみ）不具合ID・内容
- 環境：{開発 / 検証 / 本番相当}
- 実施者 ／ 由来：
```

## 4. 網羅の考え方（2系統 × カバレッジ）

**2系統を別々に網羅**することで、「動くが要望を満たさない」「要望は満たすがバグがある」の両方を防ぐ。

| 系統 | 問い | 網羅の目安 |
|------|------|-----------|
| **要件視点** | ユーザー要望が満たされているか | 全要件の**受け入れ条件**に保証記録があるか |
| **成果物視点** | 仕様通り動くか・バグがないか | 全機能・API・**バリデーション**・**状態遷移**・整合性に記録があるか |

- **バリデーション網羅**：情報定義の値域・バリデーション（§3）から機械的にケース導出（境界値・列挙外・必須欠落）。
- **状態遷移網羅**：機能仕様の状態遷移から、**不正遷移が拒否されること**を確認。
- 目的は「**何が未保証か**」を可視化すること。

```markdown
## 4. カバレッジ
- 要件網羅：| 要件ID | 保証QA-ID | 状態 |
- 仕様網羅：機能 / API / バリデーション / 状態遷移 / 整合性 ごとに保証QAの有無
```

## 5. 品質保証一覧（総覧・任意）

記録を1行ずつ俯瞰し、「何がいつ保証されたか」を一望する。

```markdown
| QA-ID | 実施日 | 対象（箇所） | 種別 | 観点 | 何を保証 | 結果 |
| ----- | ------ | ------------ | ---- | ---- | -------- | ---- |
```

## 記載ルール（章横断の作法）

1. **1記録に「いつ・何に対して・どのようなテストを・どういう観点で・何が保証できたか」を必ず揃える**。
2. **対象は成果物ドキュメントの箇所で指す**（要件 R-xxx／機能仕様 F-xxx／API仕様 API-xxx／情報定義 1-N）。曖昧な対象にしない。
3. **「何が保証できたか」を言語化する** — 合否だけでなく、どの受け入れ条件／仕様が満たされたかを成果物の箇所で示す。
4. **観点はカタログ（2-3）から付ける**。
5. **異常系は API仕様のエラーコード／機能仕様のエラーメッセージと対応づける**。
6. **バリデーション・状態遷移ケースは情報定義の値域・状態遷移から機械的に導く**。
7. **不合格は不具合として残し、再検証の記録も追記**（消さず蓄積する）。
8. **要件チケットの反映状況（品質保証列）と相互参照**し、トレーサビリティを閉じる。
9. **2系統（要件視点／成果物視点）の両方を網羅**する。

## 貫通線（このフォーマット群を縦に貫くもの・終点）

- **値域の貫通線（終点）**：情報定義の値域 → 機能の入力チェック → API のバリデーション → **品質保証のバリデーション保証記録**。
- **状態の貫通線（終点）**：情報定義の値域（状態）→ 機能仕様の状態遷移 → **品質保証の不正遷移拒否の保証記録**。
- **受け入れ条件の貫通線（終点）**：要件の受け入れ条件 → **品質保証の要件視点記録**（要件→品質保証の閉じ）。
- **トレーサビリティID連鎖の終点**：要件ID → 機能ID → API ID → **QA-ID**。品質保証記録が連鎖を閉じ、各成果物の箇所が「保証済み」になる。
