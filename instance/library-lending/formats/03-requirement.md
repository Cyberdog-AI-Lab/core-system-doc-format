# 要件 記載フォーマット（v0.2 たたき台）

**日付：** 2026-06-18（v0.2：決定コンテキスト・取り下げコンテキスト・反映結果を追加）
**記録者：** 羽床 ＋ Claude

> [補足] 参考にした実プロジェクト文書は無し。**整合性モデル**と初期のドキュメント分類メモから第一原理で設計した。情報定義・機能仕様・API仕様・品質保証の各フォーマットと対で使う。特定プラットフォーム手段には依存させず概念で書く。

---

## 位置づけ（整合性モデル）

- 要件は整合性モデルの**横串の起点**。1つの要件が、**開発成果物**（情報定義 → 機能仕様 → API仕様 → 実装）と**品質保証**の両方を駆動する。
- ゴールは「**ユーザーの要望が、成果物に反映され、品質保証で保証される**」までを**1チケットで追える**こと。
- **構想／コンセプトとは分離**する（構想は整合性チェーンの外で独立。要件は「受け入れ条件で検証できる」単位）。
- 各チケットに **ID・タグ・ステータス（反映状況）** を持たせ、トレーサビリティの起点にする。

## 全体構成（章立てテンプレート）

| 章 | 内容 | 備考 |
|----|------|------|
| 1. 概要 | 目的・関連ドキュメントへの参照 | |
| 2. 共通定義 | ID採番・区分/優先度・**タグ定義**・**ステータス定義（ライフサイクル＋反映状況）** | |
| 3. 要件チケット | 各チケットの記載テンプレート（受け入れ条件・反映状況・調整ログ 等） | |
| 4. 要件一覧（総覧） | チケットの一覧表 | 任意 |
> 以下、各章の書き方とコピペ用スケルトンを章番号に沿って示す。

## 1. 概要

目的＋**関連ドキュメント宣言**（情報定義／機能仕様／API仕様／品質保証への参照）。「ユーザー要望を検証可能な単位で管理し、各成果物への反映と品質保証での保証を追跡する」ことを明記。

```markdown
## 1. 概要
- 本書は {システム名} の要件を、チケット形式（ID・受け入れ条件・反映状況）で管理する。
- 関連ドキュメント：情報定義 / 機能仕様 / API仕様 / 品質保証
```

## 2. 共通定義

要件チケット全体で使う語彙・採番ルールを**冒頭に1回**定義する（各チケットで再定義しない）。

### 2-1. ID採番ルール

- 形式：`R-{連番}`（例 `R-001`、3桁ゼロ埋め）。区分で分けるなら `RF-`（機能）/`RN-`（非機能）も可。
- **一度振ったIDは不変**（タイトルや内容が変わっても再利用・付け替えしない）。
- **廃止・却下しても欠番にしない** — チケットは残し、ステータスで「廃止／却下」を示す（情報定義の「廃止を残す」と同じ作法）。

### 2-2. 区分・優先度

- **区分**：`機能要件` / `非機能要件`
- **優先度（MoSCoW）**：`Must`（必須）/ `Should`（推奨）/ `Could`（任意）/ `Won't`（今回やらない）

### 2-3. タグ定義（タグの種類と語彙）

タグは**軸（種類）を固定**し、値は下記の語彙から付ける。新しい値が要るときは**この表に追記してから**使う（勝手に増やさない）。

| タグ軸 | 値（語彙） | 必須 | 説明 |
|--------|-----------|------|------|
| 区分 | 機能要件 / 非機能要件 | 必須 | 2-2 と一致 |
| 非機能種別 | 性能 / セキュリティ / 可用性 / 運用性 / 保守性 / ユーザビリティ / 互換性 | 非機能要件のみ必須 | ISO/IEC 25010 を参考 |
| 領域 | （プロジェクト定義：例 予約 / 支払 / マスタ …） | 任意 | 業務領域。情報定義のエンティティ群と対応づくと良い |
| 発生源 | 顧客要望 / 法令・規制 / 運用 / 技術制約 / 内部改善 | 任意 | なぜこの要件が生まれたか |

### 2-4. ステータス定義

**(a) ライフサイクル**（チケット全体の状態。1つだけ持つ）

`起票` → `承認待ち` → `承認済` → `対応中` → `検証中` → `完了` ／ 例外：`保留` ／ `却下` ／ `廃止`

> `保留`／`却下`／`廃止` にする場合は、チケットに**取り下げコンテキスト**（理由・判断者・日付・代替）を必ず残す（3章参照）。**起票した要件が必ず実装されるとは限らない**ため、取り下げた事実と経緯も資産として保持する。

**(b) 反映状況**（要件が各層へ**どこまで反映されたか**。層ごとに状態を持つ）

| 層 | 状態の値 |
|----|----------|
| 情報定義 / 機能仕様 / API仕様 / アーキテクチャ / 実装 | `未` / `反映中` / `済` |
| 品質保証 | `未` / `反映中`（＝検証実施中） / `済`（＝検証済み） |

- 状態の言葉は**全文書共通で「未／反映中／済」の3語だけ**（要件トレーサビリティのマトリクスと同じ。言葉を増やさない）。
- **該当しない層は `−`（対象外）**と書く。目安：機能要件はアーキテクチャ層が `−`、非機能要件（性能・可用性など）は情報定義〜API仕様が `−` でアーキテクチャ層に反映する。

> **完了の定義**：受け入れ条件がすべて満たされ、かつ**該当する全層**（`−` を除く）が `済` になった状態＝ライフサイクル `完了`。

**スケルトン（共通定義の章）**

```markdown
## 2. 共通定義
### 2-1. ID採番：R-{連番}（不変／廃止も欠番にしない）
### 2-2. 区分：機能/非機能 ／ 優先度：Must/Should/Could/Won't
### 2-3. タグ定義：| タグ軸 | 値（語彙） | 必須 | 説明 |
### 2-4. ステータス：ライフサイクル（起票→…→完了/保留/却下/廃止）＋ 反映状況（情報定義/機能仕様/API仕様/アーキテクチャ/実装/品質保証 ごとに 未/反映中/済、対象外は −）
```

## 3. 要件チケット

1要件＝1チケット＝1ID。各チケットを次の項目で書く。

| 項目 | 内容 |
|------|------|
| ID | `R-xxx`（不変） |
| タイトル | 一言で何の要件か |
| 区分 / 優先度 | 機能/非機能 ／ Must など |
| タグ | 2-3 の軸:値（例 `非機能種別:性能` `領域:予約`） |
| ステータス | ライフサイクル（2-4-a） |
| 背景・目的 | **なぜ**この要件が必要か（誰のどんな課題を解くか） |
| **コンテキスト（決定の背景）** | 決定に至った状況・経緯、関係する要素・制約。**特に「誰が・誰と・どんな目的で」決めたか**を明記 |
| 要件詳細 | 何を満たすか（システムに求めること） |
| **受け入れ条件** | **検証可能な条件**のチェックリスト（→ 品質保証の要件視点ケースに 1:1） |
| **反映状況** | 各層の反映先ID・状態・**反映内容**（下表。トレーサビリティの起点） |
| **反映結果（サマリ）** | 最終的に**どの機能に・どういう内容で**反映されたか（1〜数文） |
| **取り下げコンテキスト** | `却下`/`廃止`/`保留` にした場合の**理由・判断者・日付・代替**（チケットは消さず残す） |
| スコープ外 | この要件で**やらないこと**（境界） |
| 関連要件 | 依存・親子（`R-xxx`） |
| 調整ログ | 日付｜**参加者（誰が誰と）**｜内容｜決定 |
| 起票者 / 起票日 / 最終更新日 | |

**受け入れ条件の書き方**：曖昧な形容詞を避け、**数値・条件**で。`Given / When / Then` 形式か「〜できること／〜であること」の箇条書き。非機能は**測定可能な基準**（例：応答時間 95%ile < 1秒）。

**反映状況テーブル**（反映先ID＋状態＋**反映内容**）：

| 層 | 反映先ID | 状態 | 反映内容（何をどう） |
|----|----------|------|----------------------|
| 情報定義 | 1-N | 済 / 反映中 / 未 / − | 追加/変更したエンティティ・属性・ルール |
| 機能仕様 | F-xxx | 〃 | どの画面・機能にどう |
| API仕様 | API-xxx | 〃 | どのAPIにどう |
| アーキテクチャ | ADR-xxx・§番号 | 〃 | 基盤・非機能でどう実現したか（主に非機能要件） |
| 実装 | （PR番号・コミット等） | 〃 | 実装の要点 |
| 品質保証 | QA-xxx | 〃 | 検証したケース |

**コピペ用スケルトン（チケット1枚）**

```markdown
### R-{番号} {タイトル}
- 区分 / 優先度：機能要件 / Must
- タグ：区分:機能要件, 領域:{...}（非機能なら 非機能種別:{...}）
- ステータス：承認済
- 背景・目的：
- コンテキスト（決定の背景）：誰が・誰と・どんな目的で決めたか／状況・経緯／関係要素・制約
- 要件詳細：
- 受け入れ条件：

- 反映状況：
```

## 記載ルール（章横断の作法）

1. **1要件＝1チケット＝1ID（不変）**。粒度は「受け入れ条件で検証できる」単位に切る。
2. **受け入れ条件は必ず検証可能に書く** — 曖昧な形容詞を避け数値・条件で（品質保証で確認できる形）。→ 品質保証の**要件視点ケースに 1:1** で対応。
3. **反映状況で「要件 → 各成果物 → 品質保証」のトレーサビリティを維持** — 反映先ID（情報定義/機能/API/品質保証）を必ず書く。
4. **タグは定義済みの軸・語彙から付ける**（増やすなら 2-3 に追記してから）。
5. **非機能要件は「非機能種別タグ＋測定可能な基準」を必須**にする。
6. **スコープ外を明記**して要件の境界を締める。
7. **調整ログを日付つきで残す**（なぜその受け入れ条件になったかの経緯）。
8. **廃止・却下はチケットを消さず、ステータスで残す**（ID欠番にしない）。
9. **構想／コンセプトとは混ぜない**（構想は独立。要件は検証可能な単位）。
10. **決定コンテキストを残す** — 「**誰が・誰と・どんな背景/目的で**決めたか」を明記し、後から「なぜこの判断をしたか」を辿れるようにする。
11. **取り下げも記録する** — 起票した要件が必ず実装されるとは限らない。`却下`/`廃止`/`保留` 時は**取り下げコンテキスト**（理由・判断者・日付・代替）を残し、チケットは消さない。
12. **反映結果を残す** — 反映先IDだけでなく「**どの機能に・どういう内容で**」反映されたか（反映内容・反映結果サマリ）を残す。

## 貫通線（このフォーマット群を縦に貫くもの）

- **要件が全体を挟む起点**：要件 →（成果物チェーン）情報定義 → 機能仕様 → API仕様 → 実装、要件 → 品質保証。
- **受け入れ条件の貫通線**：要件の受け入れ条件 → 品質保証の**要件視点ケース**（ユーザー要望が満たされるかの保証）。
- **トレーサビリティID連鎖の起点**：**要件ID** → 機能ID（利用画面/機能） → API ID → 品質保証ID。反映状況テーブルがこの連鎖を1枚で保持する。
- **非機能の貫通線**：非機能種別タグ（性能・セキュリティ等）→ システムアーキテクチャ§6の実現方針（反映状況の「アーキテクチャ」層で追跡）→ 品質保証の非機能観点。
