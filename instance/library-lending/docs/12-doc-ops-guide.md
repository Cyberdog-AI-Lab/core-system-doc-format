# ドキュメント運用プロセスガイド — 図書貸出システム（プロジェクト適用版）

> 本書は[運用ガイドフォーマット](../formats/12-doc-ops-guide.md)を本プロジェクトへ適用したもの。
> プロセスの原本は標準（フォーマット）側。ここでは本プロジェクト固有の適用だけを記す。

## 5. 変更フロー（本プロジェクト適用）

1. 変更は必ず[要件](03-requirement.md) R-xxx として起票（要件外の微修正は[トレーサビリティ](10-traceability.md)「要件外の改訂ログ」へ）。
2. 上流→下流の順（用語集→情報定義→機能仕様→API仕様→実装→品質保証）で反映。
3. 反映のたびに[トレーサビリティ](10-traceability.md)の「要件×段階」表の日付を更新（＝ダッシュボードのタイムラインが動く）。

## 6. AI主体メンテナンス

- 反映・整合チェックは **AIを主体**とし、人はレビューと承認を担う。
- 整合性チェックは同梱の `consistency-check` スキルで実施（下記 §8 の観点）。

## 7. 不変ルール（このプロジェクトでも順守）

- 改訂履歴は各文書に持たず[トレーサビリティ](10-traceability.md)へ一元化。
- 値域・プロパティ名の源は[用語集](02-glossary.md)（源に1回・下流は参照）。
- 廃止した要件・仕様は消さず「取り下げ」として残す。
- プラットフォーム固有手段は[アーキ](04-architecture.md) §3 に閉じ込め、他文書は概念で書く。

## 8. 整合性レビュー観点（consistency-check が検査）

- A 値域貫通：用語集の値仕様が情報定義→機能→API→品質保証まで一致。
- B 状態貫通：`CopyStatus`/`LoanStatus` の遷移が情報定義↔機能仕様↔品質保証で整合。
- C トレースID連鎖：R-xxx→F-xxx→API-xxx→QA-xxx が途切れていない。
- D 反映状況↔実体：[トレーサビリティ](10-traceability.md)の日付と各文書の実記述が一致（例：品質保証済なのにQA記録が無い、の検出）。
- F 用語集ドリフト：文書中のプロパティ名が用語集の標準名と一致。
