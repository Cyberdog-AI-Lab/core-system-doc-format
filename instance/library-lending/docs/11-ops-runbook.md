## 3. 監視・アラート対応

| アラート | 意味（何が起きている） | 一次対応の手順 | 直らないとき |
| -------- | ---------------------- | -------------- | ------------ |
| B-001 失敗 | 延滞の検出・通知が行われていない | 1. ジョブログを確認 2. 再実行（冪等なので安全＝[機能仕様](06-func-spec.md) §15） | 再失敗なら二次対応へ（当日中） |
| 応答遅延 | 蔵書検索が1秒超（NFR-01違反） | 1. 対象時間帯・画面を記録 2. 再現確認 | 二次対応へ |
| DB接続エラー | 貸出登録ができない（業務停止） | 即・障害レベル「重大」として §4 へ | — |

## 4. 障害対応

### 4-1. 障害レベル

| レベル | 定義 | 目標復旧時間 | 通知範囲 |
| ------ | ---- | ------------ | -------- |
| 重大 | 貸出・返却の登録ができない（窓口業務が止まる） | 4時間 | 責任者まで即時 |
| 中 | 検索・延滞一覧など一部機能が使えない（貸出は紙で代替可） | 当日中 | 開発担当まで |
| 軽微 | 表示くずれ等・業務は回る | 次リリース | チケット起票のみ |

### 4-2. 初動（共通）

1. 事象の記録（いつ・何が・誰に影響）
2. レベル判定（4-1）→ 通知
3. 回避策の確認（重大時：紙台帳での暫定運用に切替）
4. 復旧作業（必要なら §5 リストア・§6 切り戻し）

### 4-3. ふりかえり

復旧後に必ず記録：原因／対応の時系列／再発防止（必要なら[要件](03-requirement.md)チケット起票）。

## 6. リリース手順

### 6-1. リリース前チェックリスト

- [ ] 含む要件の反映状況が該当全層「済」（[トレーサビリティ](10-traceability.md) §2）
- [ ] 随伴移行（MIG-xxx）のリハーサル済み
- [ ] 切り戻し手順の準備（直前バックアップ＋旧版に戻す手順）
- [ ] 館内へ告知済み（実施日時・影響）

### 6-2. 手順

1. 実施時間帯：閉館後（平日夜間） 2. バックアップ取得（§5） 3. デプロイ 4. データ移行（該当時） 5. 動作確認（検索・貸出・返却の基本動線） 6. 台帳に記録（§7）

### 6-3. 切り戻し

判断基準：動作確認で基本動線が通らないとき。手順：直前バックアップへ復元＋旧版へ戻す。
